# Synchronized 锁与锁升级策略：从对象头到操作系统的全链路拆解

## 一、定义与本质：不只是"互斥"，而是一套自适应同步状态机

在大多数教科书式的定义中，`synchronized` 被简单描述为"Java 内置的互斥锁关键字"。这种定义过于表面。从架构师视角审视，**`synchronized` 的本质是一套嵌入在 JVM 对象模型中的、基于运行时竞争反馈的自适应同步状态机**。

这套状态机的核心设计哲学是：**乐观假设，悲观兜底**。JVM 假设绝大多数锁在运行时不存在真正的多线程竞争（事实上，IBM 研究表明超过 90% 的锁在整个生命周期内只被一个线程持有），因此它将锁的默认状态设计得极其轻量——几乎零开销的偏向锁。只有当运行时反馈表明竞争确实存在时，才逐级升级为更重的同步机制。

这与 `ReentrantLock` 形成鲜明对比：`ReentrantLock` 从一开始就维护一个完整的 AQS 等待队列和 CAS 状态机，属于"一视同仁"的设计；而 `synchronized` 则是"按需付费"——你用多少竞争，JVM 就给你多少同步开销。

上图展示了 `synchronized` 的技术栈全景。从顶层的 Java 语法糖，到中间层 JVM/ART 的运行时数据结构，再到底层操作系统的线程同步原语，形成了一条完整的调用链路。

**应用层**是开发者直接接触的 API 表面，包括同步代码块 `synchronized(obj) {}` 和同步方法修饰符。这两种写法在字节码层面有本质区别：同步代码块会生成显式的 `monitorenter` 和 `monitorexit` 指令；而同步方法则是在方法访问标志（access_flags）中设置 `ACC_SYNCHRONIZED` 位，由 JVM 隐式处理。

**JVM/ART 层**是锁机制的核心实现所在。三个关键数据结构各司其职：`Mark Word` 是每个对象头中 64 位（64 位系统）的状态位图，存储锁状态和持有者信息；`Lock Record` 是栈帧中的轻量级锁记录，用于 CAS 替换 Mark Word；`ObjectMonitor` 是 HotSpot 中用 C++ 实现的重量级管程结构，包含完整的等待队列和阻塞逻辑。

**操作系统层**是最终的兜底机制。当竞争激烈到必须真正阻塞线程时，JVM 会调用 POSIX 线程库的 `pthread_mutex_lock`，而 Linux 内核则通过 `futex`（Fast Userspace Mutex）系统调用实现高效的用户态/内核态切换。

## 三、对象头 Mark Word 的位图结构

理解锁升级的前提是彻底搞清楚 Mark Word 的内存布局。在 64 位 HotSpot JVM 中，对象头的 Mark Word 占据 8 字节（64 位），其结构随锁状态动态变化：

```asm
┌─────────────────────────────────────────────────────────────────────────────┐
│                         64-bit Mark Word Layout                             │
├─────────────────────────────────────────────────────────────────────────────┤
│ Lock State       │ Bit 63-3 (61 bits)                    │ Bit 2-0 (3 bits)│
├─────────────────────────────────────────────────────────────────────────────┤
│ Unlocked         │ identity_hashcode:31 | age:4 | unused:26    │ 0  0  1    │
├─────────────────────────────────────────────────────────────────────────────┤
│ Biased           │ thread_id:54 | epoch:2 | age:4 | (biased)   │ 1  0  1    │
├─────────────────────────────────────────────────────────────────────────────┤
│ Lightweight      │ ptr_to_lock_record:62                       │ 0  0  0    │
├─────────────────────────────────────────────────────────────────────────────┤
│ Heavyweight      │ ptr_to_monitor:62                           │ 1  0       │
├─────────────────────────────────────────────────────────────────────────────┤
│ GC Marked        │ (forwarding pointer / GC metadata)          │ 1  1       │
└─────────────────────────────────────────────────────────────────────────────┘

Tag Bits Decoding:
  001 = Unlocked (无锁，可偏向)
  101 = Biased (偏向锁)
  000 = Lightweight Locked (轻量级锁)
  10  = Heavyweight Locked (重量级锁，最低2位)
  11  = GC Marked (GC 标记态)
```

这张位图结构揭示了 JVM 设计的精妙之处：**最低 2-3 位的标志位决定了整个 64 位的解释方式**。这是一种典型的"标签指针"（Tagged Pointer）技术——通过复用指针的低位（因为对象地址对齐，低位必然为 0）来存储元信息，从而在不增加额外存储开销的前提下实现多态数据结构。

**无锁状态（001）**：高位存储 identity hashcode（只有调用过 `hashCode()` 才会计算并填充）、GC 分代年龄（age，4 位，最大 15）以及未使用的保留位。这个状态是对象刚创建时的默认态，也是偏向锁撤销后可能回退到的状态。

**偏向锁状态（101）**：高位存储偏向线程的 ID（54 位，足以覆盖所有线程）、epoch（2 位，用于批量重偏向优化）和 age。注意，一旦启用偏向锁，identity hashcode 就无处可存——这正是调用 `hashCode()` 会导致偏向锁撤销的根本原因。

**轻量级锁状态（00）**：整个高 62 位变成一个指针，指向当前持有锁的线程栈帧中的 Lock Record。这个指针实现了"栈上分配"的轻量级同步，避免了堆内存分配。

**重量级锁状态（10）**：高 62 位指向堆中分配的 `ObjectMonitor` 对象。一旦升级到这个状态，就意味着 JVM 认为该锁确实存在激烈竞争，需要完整的等待队列和阻塞/唤醒机制。

## 四、锁升级状态机：单向不可逆的演进路径

锁升级是 `synchronized` 最核心的运行时优化机制。JVM 根据竞争反馈动态调整锁的实现策略，但这个过程是**单向不可逆**的——一旦升级，就不会降级（除非 GC 回收重建对象）。

### 状态转换的详细推演

**阶段零：对象创建时的初始状态选择**

当 `new Object()` 执行时，JVM 并不会立即将对象标记为"偏向某个线程"。这里存在一个关键的时间窗口：JVM 启动后的前 4 秒（由 `-XX:BiasedLockingStartupDelay` 控制，默认 4000ms），所有新创建的对象都**不会**进入偏向锁状态，而是直接从无锁态开始。

这个设计的原因是：JVM 启动阶段会执行大量的类加载和初始化，这些操作涉及高度竞争的同步（如 `ClassLoader` 的锁），如果这时启用偏向锁，会导致频繁的偏向撤销，反而降低性能。4 秒后，JVM 假设启动热身完成，开始为新对象启用偏向锁。

**阶段一：偏向锁的获取与重偏向**

对象创建后（4 秒延迟后），Mark Word 处于"匿名偏向"状态——偏向锁标志位已设置（101），但 thread_id 字段为空（全 0）。当第一个线程 Thread-A 执行 `synchronized(obj)` 时，JVM 通过一次 CAS 操作将自己的线程 ID 写入 Mark Word 的 thread_id 字段。如果 CAS 成功，偏向锁获取完成，后续 Thread-A 再次进入同一个同步块时，只需简单比对 thread_id 即可，**无需任何原子操作**。

当 Thread-A 已经退出同步块（但对象仍偏向 Thread-A），Thread-B 尝试获取锁时，会发生**重偏向**（Rebias）。JVM 会检查 Thread-A 是否存活且是否仍在同步块内。如果 Thread-A 已不在同步块内，JVM 会在全局安全点（Safepoint）撤销原偏向，然后将对象重新偏向 Thread-B。这个过程涉及 STW（Stop-The-World），但通常非常短暂。

**阶段二：偏向锁撤销与轻量级锁升级**

以下三种情况会触发偏向锁撤销，升级为轻量级锁：

1. **真正的并发竞争**：Thread-B 尝试获取锁时，Thread-A 仍在同步块内。这是最直观的竞争场景。

2. **调用 `hashCode()`**：如前所述，identity hashcode 与偏向锁的 thread_id 共用 Mark Word 的高位空间。一旦计算 hashcode，就必须找地方存储它，偏向锁状态无法容纳，因此必须撤销。

3. **调用 `wait()`/`notify()`**：这些方法需要 ObjectMonitor 的 WaitSet 队列支持，而偏向锁和轻量级锁都没有这个结构，因此会直接膨胀为重量级锁。

偏向撤销的代价在于**必须等待全局安全点**。JVM 需要暂停所有应用线程，遍历偏向线程的栈帧，找到所有指向该对象的 Lock Record，然后根据情况决定是恢复为无锁态、升级为轻量级锁还是直接膨胀为重量级锁。这个 STW 暂停虽然通常只有微秒级别，但在高并发场景下频繁触发仍会造成可观的延迟抖动。

**阶段三：轻量级锁的 CAS 自旋**

升级为轻量级锁后，JVM 在当前线程的栈帧中分配一块称为 Lock Record 的空间。Lock Record 包含两个关键字段：一是 `Displaced Mark Word`，用于备份对象原本的 Mark Word；二是 `Owner` 指针，指向锁对象。

获取轻量级锁时，JVM 执行以下操作：

1. 将对象当前的 Mark Word 复制到 Lock Record 的 Displaced Mark Word 字段
2. 通过 CAS 尝试将对象的 Mark Word 更新为指向该 Lock Record 的指针
3. 如果 CAS 成功，当前线程持有轻量级锁；如果失败，进入自旋等待

自适应自旋（Adaptive Spinning）是 JDK 6 引入的优化。JVM 会根据**同一个锁上次自旋的成功率**动态调整自旋次数：如果上次自旋成功获得了锁，JVM 会乐观地增加本次自旋次数；如果上次自旋失败，则减少甚至跳过自旋。这种启发式策略避免了固定自旋次数带来的资源浪费或响应延迟。

**阶段四：重量级锁的膨胀与 ObjectMonitor**

当自旋超过阈值仍未获得锁，或者检测到已有线程在等待，JVM 判定当前存在真正的多线程竞争，触发锁膨胀（Inflation）。膨胀过程中，JVM 在堆中分配一个 `ObjectMonitor` 对象，将对象的 Mark Word 替换为指向该 Monitor 的指针（低 2 位标记为 10）。

`ObjectMonitor` 是 HotSpot 中用 C++ 实现的核心同步结构，其内存布局如下：

```cpp
// hotspot/src/share/vm/runtime/objectMonitor.hpp (简化)
class ObjectMonitor {
  // ============ 核心字段 ============
  volatile markWord _header;      // 备份的原始 Mark Word（用于锁释放时恢复）
  
  void* volatile _object;         // 指向关联的 Java 对象
  void* volatile _owner;          // 当前持有锁的线程（Thread*）
  
  volatile int _recursions;       // 重入计数（同一线程多次 synchronized）
  
  // ============ 等待队列 ============
  ObjectWaiter* volatile _EntryList;   // 阻塞等待获取锁的线程队列
  ObjectWaiter* volatile _cxq;         // 竞争队列（Contention Queue，LIFO）
  ObjectWaiter* volatile _WaitSet;     // 调用 wait() 后进入的等待队列
  
  // ============ 统计与优化字段 ============
  volatile int _contentions;      // 当前竞争计数
  volatile int _SpinDuration;     // 自适应自旋参数
  
  // ============ 底层同步原语 ============
  ParkEvent* _EntryEvent;         // 用于 park/unpark 的事件对象
};
```

这个结构体揭示了重量级锁的完整机制：`_owner` 字段记录当前持锁线程；`_recursions` 支持可重入语义（同一线程多次进入同步块）；三个队列 `_cxq`、`_EntryList`、`_WaitSet` 分别承担不同的职责。

**三大队列的协作机制**是面试高频考点，值得详细展开：

**`_cxq`（Contention Queue）**：这是一个 LIFO（后进先出）栈结构。当线程竞争锁失败时，会被封装成 `ObjectWaiter` 节点并通过 CAS 压入 `_cxq` 栈顶。使用 LIFO 而非 FIFO 的原因是：CAS 操作只能原子地修改栈顶指针，无法原子地追加到队尾。

**`_EntryList`**：这是一个 FIFO 队列，存放"即将被唤醒去竞争锁"的线程。当持锁线程释放锁时，会将 `_cxq` 中的所有节点批量迁移到 `_EntryList`，然后从 `_EntryList` 头部取出一个线程进行唤醒。这种设计将"入队"和"出队"的竞争点分离到两个不同的数据结构，减少了 CAS 冲突。

**`_WaitSet`**：存放调用了 `wait()` 方法的线程。这些线程已经释放了锁，进入等待状态，直到被 `notify()`/`notifyAll()` 唤醒。被唤醒的线程不会直接获得锁，而是被移入 `_EntryList`，重新参与锁竞争。

## 五、落地场景与避坑指南

### 场景一：偏向锁在高并发容器中的性能陷阱

```kotlin
// 典型的性能反模式
class CounterService {
    private val lock = Any()
    private var count = 0
    
    // 多线程高频调用
    fun increment() {
        synchronized(lock) {  // lock 对象频繁在不同线程间切换
            count++
        }
    }
}
```

这段代码在低并发场景下工作良好，但在高并发（多线程频繁交替获取锁）场景下会遭遇严重的性能问题。原因是：`lock` 对象的偏向锁会被频繁撤销，每次撤销都需要 STW 到达安全点。

**优化方案**：

2. **使用 `java.util.concurrent` 原子类**：`AtomicInteger` 使用无锁 CAS 操作，完全避开 `synchronized` 的锁升级开销。

3. **锁分段/分离**：参考 `ConcurrentHashMap` 的设计，将单一锁拆分为多个细粒度锁，降低竞争密度。

### 场景二：`hashCode()` 触发偏向锁撤销的隐蔽 Bug

```kotlin
// 看似无害的日志语句
fun processWithLock(obj: Any) {
    Log.d("DEBUG", "Processing object: ${obj.hashCode()}")  // 触发 hashCode 计算
    
    synchronized(obj) {
        // 业务逻辑
        // 此时 obj 已经不可能处于偏向锁状态
    }
}
```

上述代码中，在进入 `synchronized` 块之前调用了 `hashCode()`，导致对象的 Mark Word 被写入了 identity hashcode，偏向锁标志位被清除。即使后续只有单线程访问该同步块，也无法享受偏向锁的零开销优势。

**更隐蔽的变体**：将对象作为 `HashMap` 或 `HashSet` 的 Key 时，容器会调用 `hashCode()` 方法。如果这些对象后续还会被用作同步锁，偏向锁优化将完全失效。

### 6.3 为什么锁升级是单向不可逆的？

这是面试中区分度极高的问题。很多候选人能背出"升级不可逆"的结论，但无法解释原因。

**原因一：降级的收益不足以覆盖成本**

假设某个对象从重量级锁"降级"回偏向锁，JVM 需要执行以下操作：

1. 等待所有等待队列中的线程被唤醒并离开
2. 销毁或回收 `ObjectMonitor` 对象
3. 重新构建 Mark Word 的偏向锁状态
4. 处理可能的 hashcode 冲突（如果之前调用过 `hashCode()`）

这些操作的开销远超偏向锁带来的收益。更关键的是，一旦对象经历过高竞争，统计规律表明它**大概率会再次经历高竞争**，降级后很快又要升级，形成无意义的振荡。

**原因二：Mark Word 的信息不可逆丢失**

**原因三：简化实现，减少 Bug 风险**

锁机制是并发编程的基石，任何 Bug 都可能导致死锁、数据竞争等灾难性后果。单向状态机的实现远比双向状态机简单可靠。JVM 工程师选择了"宁可牺牲理论上的极致性能，也要确保实现的正确性和可维护性"。

## 七、monitorenter/monitorexit 字节码指令深度解析

`synchronized` 代码块在编译后会生成 `monitorenter` 和 `monitorexit` 字节码指令。这两条指令是理解锁实现的关键入口。

### 7.1 字节码层面的指令结构

```java
// Java 源码
public void syncMethod() {
    synchronized (this) {
        doSomething();
    }
}
```

编译后的字节码（通过 `javap -c` 查看）：

```java

     1: dup                        // 复制栈顶（用于后续的 monitorexit）

     3: monitorenter               // 【关键】获取锁
     4: aload_0
     5: invokevirtual #2           // 调用 doSomething()
     8: aload_1
     9: monitorexit                // 【关键】正常退出释放锁
    10: goto 18                    // 跳转到方法返回
    13: astore_2                   // 异常处理器：存储异常
    14: aload_1
    15: monitorexit                // 【关键】异常退出也要释放锁！
    16: aload_2

     from    to  target type

```

上述字节码揭示了一个关键设计：**编译器会自动生成两个 `monitorexit` 指令**——一个用于正常执行路径（第 9 行），一个用于异常处理路径（第 15 行）。这确保了无论代码是正常返回还是抛出异常，锁都会被正确释放，**从语言层面杜绝了忘记释放锁的可能**。

### 7.2 monitorenter 指令的 JVM 实现路径

`monitorenter` 指令在 HotSpot JVM 中的执行路径是一个典型的"快速路径 + 慢速路径"分层设计。JVM 首先尝试开销最低的获取方式，只有失败时才逐级回退到更重的机制。

```mermaid
sequenceDiagram
    autonumber
    participant T as 当前线程
    participant MW as Mark Word
    participant LR as Lock Record (栈帧)
    participant OM as ObjectMonitor
    participant OS as 操作系统
    
    T->>MW: 读取锁状态位
    
    alt 偏向锁路径 (101)
        MW-->>T: 检查 thread_id
        alt thread_id == 当前线程
            T->>T: 直接进入 (零开销)
        else thread_id == 0 (匿名偏向)
            T->>MW: CAS 写入 thread_id
            MW-->>T: 成功则进入
        else thread_id == 其他线程
            T->>T: 触发偏向撤销 (STW)
            T->>LR: 进入轻量级锁路径
        end
    end
    
    alt 轻量级锁路径 (00)
        T->>LR: 分配 Lock Record
        T->>LR: 拷贝 Mark Word → Displaced Header
        T->>MW: CAS (Mark Word → LR指针)
        alt CAS 成功
            MW-->>T: 获取锁成功
        else CAS 失败
            T->>T: 自适应自旋
            alt 自旋成功
                T->>T: 重试 CAS
            else 自旋失败
                T->>OM: 锁膨胀
            end
        end
    end
    
    alt 重量级锁路径 (10)
        T->>OM: 尝试 CAS 设置 _owner
        alt CAS 成功
            OM-->>T: 获取锁成功
        else CAS 失败
            T->>OM: 入队 _cxq
            T->>OS: park() 阻塞
            OS-->>T: unpark() 唤醒
            T->>OM: 重新竞争
        end
    end
```

上图展示了 `monitorenter` 的完整执行时序。每一步的决策都基于 Mark Word 的当前状态位，形成了一个三层的 fallback 机制。

**偏向锁快速路径**的核心是 thread_id 比对。当 Mark Word 中的 thread_id 等于当前线程 ID 时，JVM 仅执行一次普通的内存读取和比较操作，无需任何原子指令。这就是偏向锁"几乎零开销"的秘密——它把最常见的"同一线程重复获取锁"场景优化到了极致。

**轻量级锁路径**的核心是 Lock Record 和 CAS。JVM 在当前栈帧中分配一个 Lock Record 结构，将对象当前的 Mark Word 备份到 Lock Record 的 `Displaced Header` 字段，然后尝试用 CAS 将对象的 Mark Word 替换为指向该 Lock Record 的指针。这里的精妙之处在于：Lock Record 是栈上分配的，无需堆内存分配，且天然与线程绑定。

**重量级锁路径**是最终的兜底机制。当 CAS 自旋失败后，JVM 创建或复用一个 `ObjectMonitor` 对象，将当前线程封装为 `ObjectWaiter` 节点加入 `_cxq` 队列，然后调用 `park()` 陷入阻塞。

**Displaced Mark Word** 的作用是双重的：首先，它备份了对象原始的 Mark Word 内容，用于锁释放时恢复；其次，它作为"我是锁持有者"的凭证——只有 Mark Word 指向自己 Lock Record 的线程才能成功执行 `monitorexit`。

这种基于栈的重入计数方式比 `ReentrantLock` 的整数计数器更优雅：它天然支持异常场景下的正确释放（每个 `monitorexit` 只负责"自己的" Lock Record），且无需额外的原子操作来更新计数器。

### 7.4 monitorexit 指令的执行逻辑

`monitorexit` 的实现是 `monitorenter` 的镜像过程，但有一个关键的额外职责：**唤醒等待线程**。

**偏向锁的退出**是最轻量的——实际上什么都不做。偏向锁的设计假设是"同一线程会反复获取"，因此释放时不清除 thread_id，下次获取时直接比对即可。

**轻量级锁的退出**需要一次 CAS 操作：将对象的 Mark Word 从"指向 Lock Record 的指针"恢复为"原始的 Mark Word"（从 `Displaced Mark Word` 字段取出）。如果 CAS 失败，说明在持锁期间发生了锁膨胀，需要切换到重量级锁的释放路径。

**重量级锁的退出**涉及队列管理和线程唤醒。释放线程将 `_owner` 设为 `NULL`，然后检查是否有线程在等待。如果有，需要将 `_cxq` 中的节点迁移到 `_EntryList`，并 `unpark()` 唤醒队首线程。被唤醒的线程不会直接获得锁，而是重新尝试 CAS 竞争——这是为了支持"非公平锁"语义，允许新来的线程抢占。

### 8.2 锁膨胀（Inflation）的实现细节

锁膨胀是从轻量级锁升级到重量级锁的关键过程。HotSpot 实现了一套精细的膨胀协议，确保并发场景下的正确性。

```cpp
// 简化的膨胀逻辑伪代码
ObjectMonitor* ObjectSynchronizer::inflate(Thread* self, oop object) {
    for (;;) {
        markWord mark = object->mark();
        
        // Case 1: 已经是重量级锁，直接返回
        if (mark.has_monitor()) {
            return mark.monitor();
        }
        
        // Case 2: 正在被其他线程膨胀，自旋等待
        if (mark == markWord::INFLATING()) {
            // 短暂自旋，避免立即阻塞
            SpinPause();
            continue;
        }
        
        // Case 3: 轻量级锁状态，需要膨胀
        if (mark.has_locker()) {
            // 分配新的 ObjectMonitor
            ObjectMonitor* monitor = allocate_monitor(self);
            
            // CAS 设置膨胀中标记
            markWord cmp = object->cas_set_mark(markWord::INFLATING(), mark);
            if (cmp != mark) {
                // CAS 失败，释放 monitor 重试
                free_monitor(monitor);
                continue;
            }
            
            // 膨胀成功，初始化 monitor
            monitor->set_header(mark.displaced_mark_helper());
            monitor->set_owner(mark.locker());  // 原持有者
            monitor->set_object(object);
            
            // 原子地将对象头指向 monitor
            object->release_set_mark(markWord::encode(monitor));
            return monitor;
        }
        
        // Case 4: 无锁状态，直接膨胀
        ObjectMonitor* monitor = allocate_monitor(self);
        monitor->set_header(mark);
        
        markWord cmp = object->cas_set_mark(markWord::encode(monitor), mark);
        if (cmp == mark) {
            return monitor;
        }
        // CAS 失败，释放 monitor 重试
        free_monitor(monitor);
    }
}
```

### 8.3 EnterI：重量级锁的获取实现

当线程尝试获取已膨胀的重量级锁时，会进入 `ObjectMonitor::EnterI` 方法。这是 HotSpot 中最复杂的同步原语之一。

```cpp
// 简化的 EnterI 逻辑
void ObjectMonitor::EnterI(Thread* self) {
    // 快速路径：尝试 CAS 直接获取
    if (TryLock(self) == TryLockResult::Success) {
        return;
    }
    
    // 自适应自旋
    if (TrySpin(self) == TryLockResult::Success) {
        return;
    }
    
    // 构造等待节点
    ObjectWaiter node(self);
    node.TState = ObjectWaiter::TS_CXQ;
    
    // CAS 入队到 _cxq 栈顶
    for (;;) {
        node._next = _cxq;
        if (Atomic::cmpxchg(&_cxq, node._next, &node) == node._next) {
            break;
        }
    }
    
    // 入队成功，准备阻塞
    for (;;) {
        // 阻塞前再试一次
        if (TryLock(self) == TryLockResult::Success) {
            // 成功！需要从队列中移除自己
            UnlinkAfterAcquire(self, &node);
            return;
        }
        
        // 真正阻塞
        Self->_ParkEvent->park();
        
        // 被唤醒，重试获取
        if (TryLock(self) == TryLockResult::Success) {
            UnlinkAfterAcquire(self, &node);
            return;
        }
        // 获取失败，继续循环（可能被虚假唤醒）
    }
}
```

这个实现有几个值得注意的设计细节：

**入队后再试一次（TryLock before park）**：线程在 `park()` 阻塞之前会再做一次 `TryLock()` 尝试。这是因为在入队操作和检查锁状态之间存在时间窗口，锁可能恰好在这个窗口内被释放。如果不做这次检查，线程会错过已经空闲的锁而白白阻塞。

**虚假唤醒（Spurious Wakeup）处理**：`park()` 返回并不一定意味着锁已可用——可能是操作系统的虚假唤醒，也可能是其他线程抢先获得了锁。因此 `EnterI` 在唤醒后仍需循环重试。

**_cxq 使用 LIFO 而非 FIFO**：新来的竞争者被压入 `_cxq` 栈顶而非队尾。这个反直觉的设计是为了简化并发控制——CAS 只能原子操作一个指针，操作栈顶是天然原子的，而维护队尾需要额外的同步。

### 8.4 Exit：重量级锁的释放与唤醒策略

锁释放的复杂度不亚于获取。`ObjectMonitor::exit` 需要处理重入计数、队列迁移和线程唤醒，同时还要考虑性能优化。

```cpp
// 简化的 exit 逻辑
void ObjectMonitor::exit(Thread* self) {
    // 检查重入
    if (_recursions != 0) {
        _recursions--;
        return;  // 只是重入退出，不真正释放
    }
    
    // 真正释放锁
    _owner = NULL;
    
    // 内存屏障：确保 _owner = NULL 对其他线程可见
    OrderAccess::storeload();
    
    // 检查是否有等待者
    if (_cxq == NULL && _EntryList == NULL) {
        return;  // 无等待者，直接返回
    }
    
    // 有等待者，需要唤醒
    // 重新获取锁（防止与新来的竞争者冲突）
    if (Atomic::cmpxchg(&_owner, (void*)NULL, self) != NULL) {
        return;  // 有线程已经抢到锁，不需要我们唤醒
    }
    
    // 将 _cxq 迁移到 _EntryList
    ObjectWaiter* list = _cxq;
    if (list != NULL) {
        _cxq = NULL;
        
        // 策略选择：QMode 决定如何处理队列
        if (QMode == 2) {
            // 直接唤醒 _cxq 队首（最近入队的）
            list->TState = ObjectWaiter::TS_ENTER;
            _EntryList = list;
        } else {
            // 反转链表后追加到 _EntryList（FIFO 公平性）
            ObjectWaiter* reversed = reverse_list(list);
            append_to_EntryList(reversed);
        }
    }
    
    // 唤醒 _EntryList 队首线程
    ObjectWaiter* w = _EntryList;
    if (w != NULL) {
        _EntryList = w->_next;
        w->TState = ObjectWaiter::TS_ENTER;
        w->_thread->_ParkEvent->unpark();
    }
    
    // 释放刚才临时获取的锁
    _owner = NULL;
}
```

**wait() 的关键实现细节**：

```cpp
// 简化的 wait 逻辑
void ObjectMonitor::wait(jlong millis, Thread* self) {
    // 检查当前线程是否持有锁
    CHECK_OWNER(self);
    
    // 创建等待节点
    ObjectWaiter node(self);
    node.TState = ObjectWaiter::TS_WAIT;
    
    // 加入 _WaitSet（需要持有 _WaitSetLock）
    AddWaiter(&node);
    
    // 保存当前重入计数
    intx save_recursions = _recursions;
    _recursions = 0;
    
    // 释放锁
    _owner = NULL;
    OrderAccess::storeload();
    
    // 唤醒 _EntryList 中的等待者（如果有）
    exit_for_wait(self);
    
    // 阻塞等待
    if (millis <= 0) {
        self->_ParkEvent->park();      // 无限等待
    } else {
        self->_ParkEvent->park(millis); // 超时等待
    }
    
    // 被唤醒，重新获取锁
    EnterI(self);
    
    // 恢复重入计数
    _recursions = save_recursions;
}
```

**notify vs notifyAll 的实现差异**：

* **`notify()`**：从 `_WaitSet` 中取出**一个**节点，将其状态改为 `TS_ENTER`，移入 `_EntryList`。HotSpot 的默认策略是取队首（即等待时间最长的），但 JVM 规范并不保证这一点。

* **`notifyAll()`**：将 `_WaitSet` 中的**所有**节点批量移入 `_EntryList`。这些线程会同时参与锁竞争，只有一个能成功，其余继续阻塞在 `_EntryList` 中。

**一个常见的误解**：很多人认为 `notify()` 会立即唤醒等待线程。实际上，`notify()` 只是将线程从 `_WaitSet` 移到 `_EntryList`，真正的 `unpark()` 唤醒发生在调用 `notify()` 的线程执行 `monitorexit` 释放锁之后。

## 九、操作系统层：futex 与 park/unpark

当重量级锁需要真正阻塞线程时，最终会调用到操作系统的同步原语。在 Linux 上，这个原语是 **futex**（Fast Userspace Mutex）。

futex 的核心思想是：**在无竞争时完全在用户态完成，只有真正需要阻塞时才陷入内核**。这与 `synchronized` 的锁升级理念一脉相承——都是基于"大多数情况下无竞争"的乐观假设。

### 9.2 park/unpark 的实现

HotSpot 在 futex 之上封装了 `os::PlatformEvent`，提供 `park()` 和 `unpark()` 接口：

```cpp
// linux 实现简化版
void os::PlatformEvent::park() {
    int status;
    
    // 快速检查：如果已经有 unpark 信号，直接返回
    if (Atomic::xchg(&_event, 0) > 0) {
        return;
    }
    
    // 进入阻塞循环
    while (_event <= 0) {
        status = futex_wait(&_event, 0, NULL);  // 阻塞直到 _event 变为非 0
        
        // 处理中断
        if (status == -EINTR) {
            continue;  // 被信号中断，继续等待
        }
    }
    
    // 消费信号
    _event = 0;
}

void os::PlatformEvent::unpark() {
    // 设置信号
    if (Atomic::xchg(&_event, 1) < 1) {
        // 之前是 0 或负数，说明有线程在等待
        futex_wake(&_event, 1);  // 唤醒一个等待者
    }
}
```

**park/unpark 的许可证（permit）语义**是一个重要特性：`unpark()` 可以在 `park()` 之前调用，相当于"预存"一个唤醒信号。这避免了经典的"先唤醒后等待"（lost wakeup）问题，使得 `synchronized` 的实现更加健壮。

## 十、性能与权衡：量化分析锁的开销

理解锁机制的工程价值，需要从定量角度分析各级锁的性能特征。以下数据基于典型的 x86-64 服务器环境（Intel Xeon，Linux 5.x），Android/ARM 平台会有差异但数量级相近。

从这张表可以得出几个关键结论：

**偏向锁的价值在于"命中"场景**：当同一线程反复获取同一把锁时，偏向锁的开销几乎与普通内存访问相当。但如果频繁发生偏向撤销，10,000+ ns 的 STW 开销会严重拖累性能。

**轻量级锁的自旋是一把双刃剑**：自旋成功意味着避免了系统调用，100-500 ns 的开销远低于线程切换。但自旋失败则浪费了 CPU 时间，在高竞争场景下会加剧性能恶化。

**重量级锁的开销主要来自上下文切换**：单纯的 `ObjectMonitor` 操作只需 ~100 ns，但一旦涉及线程阻塞/唤醒，上下文切换的开销会占据主导地位。

**偏向锁**几乎没有额外内存开销——线程 ID 直接存储在对象头的 Mark Word 中，无需分配任何辅助结构。

**选型建议**：

* **优先使用 `synchronized`**：对于大多数场景，`synchronized` 的锁升级机制提供了更好的无竞争性能，且编译器自动处理锁释放，更不容易出错。

* **使用 `ReentrantLock` 的场景**：

  * 需要 `tryLock()` 非阻塞尝试

### 12.3 偏向锁撤销导致的延迟毛刺

偏向锁撤销需要 STW（Stop-The-World），虽然单次暂停时间通常很短（微秒级），但在以下场景可能造成可感知的延迟毛刺：

```kotlin
// 场景复现：高频偏向撤销
class BiasRevocationDemo {
    private val locks = Array(1000) { Any() }  // 1000 个锁对象
    
    // 线程 A：顺序访问所有锁
    fun threadA() {
        repeat(1000) { i ->
            synchronized(locks[i]) {
                // 首次访问，建立偏向
                doWork()
            }
        }
    }
    
    // 线程 B：同样顺序访问所有锁
    fun threadB() {
        repeat(1000) { i ->
            synchronized(locks[i]) {
                // 触发偏向撤销！每个锁都要 STW
                doWork()
            }
        }
    }
    
    // 问题：如果 A、B 交替执行，会触发 1000 次偏向撤销
    // 每次撤销 ~10μs STW，累计 ~10ms 全局暂停
}
```

**批量重偏向（Bulk Rebias）与批量撤销（Bulk Revoke）**是 JVM 对此类场景的优化策略：

* **批量重偏向**：当某个类的实例发生多次偏向撤销（超过 `BiasedLockingBulkRebiasThreshold`，默认 20）后，JVM 会认为该类的对象存在多线程交替访问的模式，触发批量重偏向——将该类所有实例的 epoch 值加 1，使旧的偏向失效，允许重新偏向新线程。

* **批量撤销**：当批量重偏向后仍然频繁发生撤销（超过 `BiasedLockingBulkRevokeThreshold`，默认 40），JVM 判定该类的对象不适合偏向锁优化，会禁用该类所有实例的偏向锁能力，直接从轻量级锁开始。

```kotlin
class WrongUsage1 {
    private val lock = Any()
    
    fun wrongWait() {
        lock.wait()  // ❌ IllegalMonitorStateException
    }
    
    fun correctWait() {
        synchronized(lock) {
            lock.wait()  // ✓ 正确
        }
    }
}

// 错误示例 2: synchronized 和 wait 的对象不一致
class WrongUsage2 {
    private val lock1 = Any()
    private val lock2 = Any()
    
    fun wrongWait() {
        synchronized(lock1) {
            lock2.wait()  // ❌ 持有 lock1，但 wait 在 lock2 上
        }
    }
    
    fun correctWait() {
        synchronized(lock2) {
            lock2.wait()  // ✓ 持有的锁和 wait 的对象一致
        }
    }
}

// 错误示例 3: 将 ReentrantLock 与 Object.wait() 混用
class WrongUsage3 {
    private val reentrantLock = ReentrantLock()
    private val condition = reentrantLock.newCondition()
    private val legacyLock = Any()
    
    fun wrongMix() {
        reentrantLock.lock()
        try {
            legacyLock.wait()  // ❌ ReentrantLock 不等于 synchronized
        } finally {
            reentrantLock.unlock()
        }
    }
    
    fun correctUsage() {
        reentrantLock.lock()
        try {
            condition.await()  // ✓ 使用 Condition 而非 Object.wait()
        } finally {
            reentrantLock.unlock()
        }
    }
}
```

### 15.1 自定义锁监控工具

在生产环境中，监控锁的竞争情况是性能调优的基础。以下是一个基于 JMX 的锁监控工具实现：

```kotlin
import java.lang.management.ManagementFactory
import java.lang.management.ThreadInfo
import java.lang.management.ThreadMXBean
import java.util.concurrent.ConcurrentHashMap
import java.util.concurrent.atomic.AtomicLong

/**
 * 锁竞争监控器
 * 通过定期采样线程状态，统计锁等待情况
 */
object LockContentionMonitor {
    
    private val threadMXBean: ThreadMXBean = ManagementFactory.getThreadMXBean()
    
    // 锁对象 -> 等待次数统计
    private val contentionStats = ConcurrentHashMap<String, AtomicLong>()
    
    // 锁对象 -> 累计等待时间 (ns)
    private val waitTimeStats = ConcurrentHashMap<String, AtomicLong>()
    
    init {
        // 启用线程竞争监控（可能有性能开销）
        if (threadMXBean.isThreadContentionMonitoringSupported) {
            threadMXBean.isThreadContentionMonitoringEnabled = true
        }
    }
    
    /**
     * 采样当前所有线程的锁等待状态
     */
    fun sample() {
        val threadIds = threadMXBean.allThreadIds
        val threadInfos = threadMXBean.getThreadInfo(threadIds, 10)
        
        threadInfos.filterNotNull().forEach { info ->
            if (info.threadState == Thread.State.BLOCKED) {
                val lockInfo = info.lockInfo
                if (lockInfo != null) {
                    val lockKey = "${lockInfo.className}@${lockInfo.identityHashCode}"
                    
                    // 统计等待次数
                    contentionStats.computeIfAbsent(lockKey) { AtomicLong() }
                        .incrementAndGet()
                    
                    // 统计等待时间
                    val blockedTime = info.blockedTime
                    if (blockedTime > 0) {
                        waitTimeStats.computeIfAbsent(lockKey) { AtomicLong() }
                            .addAndGet(blockedTime * 1_000_000) // ms -> ns
                    }
                }
            }
        }
    }
    
    /**
     * 获取竞争最激烈的 Top N 锁
     */
    fun getTopContentedLocks(n: Int): List<LockContentionInfo> {
        return contentionStats.entries
            .map { (lock, count) ->
                LockContentionInfo(
                    lockIdentity = lock,
                    contentionCount = count.get(),
                    totalWaitTimeNs = waitTimeStats[lock]?.get() ?: 0
                )
            }
            .sortedByDescending { it.contentionCount }
            .take(n)
    }
    
    /**
     * 检测死锁
     */
    fun detectDeadlock(): DeadlockInfo? {
        val deadlockedIds = threadMXBean.findDeadlockedThreads() ?: return null
        
        val threadInfos = threadMXBean.getThreadInfo(deadlockedIds, true, true)
        val chains = mutableListOf<DeadlockChain>()
        
        threadInfos.filterNotNull().forEach { info ->
            chains.add(
                DeadlockChain(
                    threadName = info.threadName,
                    waitingOn = info.lockInfo?.toString() ?: "unknown",
                    heldLocks = info.lockedSynchronizers.map { it.toString() },
                    stackTrace = info.stackTrace.take(10).map { it.toString() }
                )
            )
        }
        
        return DeadlockInfo(chains)
    }
    
    data class LockContentionInfo(
        val lockIdentity: String,
        val contentionCount: Long,
        val totalWaitTimeNs: Long
    ) {
        val avgWaitTimeMs: Double
            get() = if (contentionCount > 0) {
                totalWaitTimeNs.toDouble() / contentionCount / 1_000_000
            } else 0.0
    }
    
    data class DeadlockChain(
        val threadName: String,
        val waitingOn: String,
        val heldLocks: List<String>,
        val stackTrace: List<String>
    )
    
    data class DeadlockInfo(val chains: List<DeadlockChain>)
}
```

### 15.2 锁粒度优化实战

锁粒度是影响并发性能的关键因素。以下示例展示如何从粗粒度锁逐步优化到细粒度锁：

```kotlin
/**
 * 版本 1: 粗粒度锁 - 整个数据结构一把锁
 * 问题: 所有操作串行化，无法并发
 */
class CoarseGrainedCache<K, V> {
    private val map = HashMap<K, V>()
    private val lock = Any()
    
    fun get(key: K): V? = synchronized(lock) {
        map[key]
    }
    
    fun put(key: K, value: V) = synchronized(lock) {
        map[key] = value
    }
    
    fun computeIfAbsent(key: K, compute: () -> V): V = synchronized(lock) {
        map[key] ?: compute().also { map[key] = it }
    }
}

/**
 * 版本 2: 分段锁 - 按 key 哈希分散到多个锁
 * 优势: 不同段可以并发访问
 */
class SegmentedCache<K, V>(private val segmentCount: Int = 16) {
    
    private val segments = Array(segmentCount) { Segment<K, V>() }
    
    private fun segmentFor(key: K): Segment<K, V> {
        val hash = key.hashCode()
        val index = (hash xor (hash ushr 16)) and (segmentCount - 1)
        return segments[index]
    }
    
    fun get(key: K): V? {
        return segmentFor(key).get(key)
    }
    
    fun put(key: K, value: V) {
        segmentFor(key).put(key, value)
    }
    
    private class Segment<K, V> {
        private val map = HashMap<K, V>()
        private val lock = Any()
        
        fun get(key: K): V? = synchronized(lock) { map[key] }
        fun put(key: K, value: V) = synchronized(lock) { map[key] = value }
    }
}

/**
 * 版本 3: 读写分离 - 读多写少场景优化
 * 优势: 多个读操作可以并发
 */
class ReadWriteCache<K, V> {
    private val map = HashMap<K, V>()
    private val rwLock = ReentrantReadWriteLock()
    private val readLock = rwLock.readLock()
    private val writeLock = rwLock.writeLock()
    
    fun get(key: K): V? {
        readLock.lock()
        try {
            return map[key]
        } finally {
            readLock.unlock()
        }
    }
    
    fun put(key: K, value: V) {
        writeLock.lock()
        try {
            map[key] = value
        } finally {
            writeLock.unlock()
        }
    }
}

/**
 * 版本 4: 无锁化 - 使用 ConcurrentHashMap
 * 优势: 最高并发度，细粒度锁 + CAS
 */
class LockFreeCache<K, V> {
    private val map = ConcurrentHashMap<K, V>()
    
    fun get(key: K): V? = map[key]
    
    fun put(key: K, value: V) {
        map[key] = value
    }
    
    fun computeIfAbsent(key: K, compute: () -> V): V {
        return map.computeIfAbsent(key) { compute() }
    }
}
```

### 15.3 Android 场景：主线程安全的单例模式演进

在 Android 开发中，单例模式与 `synchronized` 的结合是最常见的应用场景之一。以下展示不同实现方式的演进：

```kotlin
/**
 * 方式 1: 饿汉式 - 类加载时初始化
 * 优点: 线程安全，无锁
 * 缺点: 无法延迟初始化，可能浪费资源
 */
object EagerSingleton {
    val instance = ExpensiveResource()
}

/**
 * 方式 2: 双重检查锁 (DCL) - synchronized 经典应用
 * 优点: 延迟初始化，线程安全
 * 注意: 必须使用 @Volatile，否则存在指令重排问题
 */
class DCLSingleton private constructor() {
    companion object {
        @Volatile
        private var instance: DCLSingleton? = null
        
        fun getInstance(): DCLSingleton {
            // 第一次检查：避免不必要的同步
            if (instance == null) {
                synchronized(this) {
                    // 第二次检查：防止重复创建
                    if (instance == null) {
                        instance = DCLSingleton()
                    }
                }
            }
            return instance!!
        }
    }
}

/**
 * 方式 3: Kotlin lazy 委托 - 推荐方式
 * 内部实现: 使用 synchronized + DCL
 * 优点: 代码简洁，线程安全
 */
class LazySingleton private constructor() {
    companion object {
        val instance: LazySingleton by lazy(LazyThreadSafetyMode.SYNCHRONIZED) {
            LazySingleton()
        }
    }
}

/**
 * 方式 4: 静态内部类 - JVM 类加载保证
 * 优点: 延迟初始化 + 无锁
 * 原理: 利用 JVM 类加载的线程安全性
 */
class HolderSingleton private constructor() {
    companion object {
        fun getInstance(): HolderSingleton = Holder.INSTANCE
    }
    
    private object Holder {
        val INSTANCE = HolderSingleton()
    }
}
```

**lazy 委托的内部实现剖析**：

```kotlin
// Kotlin stdlib 中 SynchronizedLazyImpl 的简化版本
private class SynchronizedLazyImpl<out T>(initializer: () -> T) : Lazy<T> {
    private var initializer: (() -> T)? = initializer
    @Volatile private var _value: Any? = UNINITIALIZED_VALUE
    
    private val lock = Any()
    
    override val value: T
        get() {
            val v1 = _value
            // 快速路径：已初始化
            if (v1 !== UNINITIALIZED_VALUE) {
                @Suppress("UNCHECKED_CAST")
                return v1 as T
            }
            
            // 慢速路径：需要同步
            return synchronized(lock) {
                val v2 = _value
                if (v2 !== UNINITIALIZED_VALUE) {
                    @Suppress("UNCHECKED_CAST")
                    v2 as T
                } else {
                    val typedValue = initializer!!()
                    _value = typedValue
                    initializer = null  // 释放初始化器，帮助 GC
                    typedValue
                }
            }
        }
    
    override fun isInitialized(): Boolean = _value !== UNINITIALIZED_VALUE
}
```

### 15.4 避免锁的常见陷阱

```kotlin
/**
 * 陷阱 1: 锁对象逃逸
 * 问题: 外部代码可能获取同一把锁，导致意外死锁
 */
class LeakyLock {
    val lock = Any()  // ❌ public 暴露了锁对象
    
    fun doSomething() {
        synchronized(lock) {
            // 临界区
        }
    }
}

// 外部代码可能这样滥用:
fun maliciousCode(leaky: LeakyLock) {
    synchronized(leaky.lock) {  // 获取了相同的锁
        while (true) { /* 永久持有，导致 DoS */ }
    }
}

/**
 * 修复: 使用 private 锁对象
 */
class SafeLock {
    private val lock = Any()  // ✓ private，外部无法访问
    
    fun doSomething() {
        synchronized(lock) {
            // 临界区
        }
    }
}

/**
 * 陷阱 2: 对可变对象加锁
 * 问题: 锁对象被替换后，多个线程可能持有"不同的锁"
 */
class MutableLockObject {
    private var lock: Any = Any()
    
    fun resetLock() {
        lock = Any()  // ❌ 锁对象被替换！
    }
    
    fun doSomething() {
        synchronized(lock) {  // 不同调用可能锁在不同对象上
            // 临界区不再安全
        }
    }
}

/**
 * 修复: 使用 val 声明锁对象
 */
class ImmutableLockObject {
    private val lock = Any()  // ✓ val 保证不可变
    
    fun doSomething() {
        synchronized(lock) {
            // 临界区
        }
    }
}

/**
 * 陷阱 3: 对字符串字面量加锁
 * 问题: 字符串驻留导致不同位置的"相同字符串"是同一对象
 */
class StringLockTrap {
    fun methodA() {
        synchronized("LOCK") {  // ❌ 字符串字面量
            // ...
        }
    }
}

// 另一个完全不相关的类:
class UnrelatedClass {
    fun methodB() {
        synchronized("LOCK") {  // 与 methodA 用的是同一把锁！
            // 可能导致意外的相互阻塞
        }
    }
}

/**
 * 陷阱 4: 对 Integer/Long 等包装类加锁
 * 问题: -128 到 127 范围内的值会被缓存复用
 */
class BoxedLockTrap {
    private var lockId: Int = 100
    
    fun doSomething() {
        synchronized(lockId) {  // ❌ 自动装箱 -> Integer.valueOf(100)
            // Integer.valueOf 对小整数有缓存，可能与其他代码共享同一对象
        }
    }
}
```
