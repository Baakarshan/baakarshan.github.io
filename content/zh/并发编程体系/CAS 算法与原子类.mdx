# CAS 算法与原子类：无锁编程的底层基石

## 一、重新定义 CAS：从「比较并交换」到「乐观并发控制」

CAS（Compare-And-Swap）绝非简单的"比较然后交换"操作。从架构师视角审视，CAS 是一种**乐观并发控制（Optimistic Concurrency Control）** 的硬件级实现——它假设冲突是小概率事件，因此不预先加锁阻塞线程，而是在提交修改时检测冲突并在失败时重试。

CAS 操作的本质是一个**原子性三元组**：

```c
// 伪代码表达 CAS 的语义
bool CAS(V* address, T expected, T desired) {
    atomic {  // 硬件保证这整块是原子的
        if (*address == expected) {
            *address = desired;
            return true;   // 成功：内存值符合预期，已更新
        }
        return false;      // 失败：内存值已被其他线程修改
    }
}
```

这个原子性保证来自 **CPU 指令级别**，而非操作系统或 JVM。在 x86/x64 架构上，对应的是 `LOCK CMPXCHG` 指令；在 ARM 架构（Android 主战场）上，则是 `LDXR/STXR`（Load-Exclusive/Store-Exclusive）指令对配合内存屏障实现。

**CAS 与 synchronized 的本质差异**：

| 维度    | synchronized（悲观锁） | CAS（乐观锁）     |
| ----- | ----------------- | ------------ |
| 假设前提  | 冲突一定会发生           | 冲突很少发生       |
| 阻塞行为  | 获取不到锁则阻塞挂起        | 永不阻塞，失败则自旋重试 |
| 上下文切换 | 可能触发内核态切换         | 纯用户态操作       |
| 适用场景  | 临界区耗时长、竞争激烈       | 临界区极短、竞争不激烈  |

**应用层（绿色区域）** 是开发者直接接触的 API 层。当你在 Android 应用中使用 `AtomicInteger` 进行线程安全的计数，或者在协程调度器中维护任务队列的头尾指针时，你实际上是在使用 CAS 的上层封装。这一层的设计目标是**易用性**——屏蔽底层复杂度，提供语义清晰的原子操作接口。

**Framework 层（蓝色区域）** 是 Java 并发包的核心所在。`java.util.concurrent.atomic` 包提供了各种原子类；`AbstractQueuedSynchronizer`（AQS）是 `ReentrantLock`、`Semaphore`、`CountDownLatch` 等同步器的骨架，其核心状态变更完全依赖 CAS；`ConcurrentHashMap` 在 JDK 8+ 的实现中，使用 CAS 配合 synchronized 实现高并发下的线程安全；`LongAdder` 则是针对高竞争场景的优化方案，通过"分段累加 + 最终汇总"的思路降低 CAS 的冲突概率。**这一层的所有机制，本质上都是对 CAS 的上层抽象与组合**。

**Native/ART 层（橙色区域）** 是 Java 与硬件之间的桥梁。`sun.misc.Unsafe` 类（在 Android 上通过隐藏 API 或 VarHandle 访问）提供了 `compareAndSwapInt`、`compareAndSwapObject` 等方法，这些方法被声明为 `native`，由 ART 虚拟机通过 JNI 调用底层实现。ART 负责插入必要的**内存屏障（Memory Barrier）**，确保 CAS 操作的可见性语义在弱内存模型的 ARM 处理器上也能正确执行。

**硬件层（橙色底部区域）** 是 CAS 的最终执行者。x86 架构使用 `LOCK CMPXCHG` 指令，通过**总线锁或缓存锁**保证原子性；ARM 架构使用 `LDXR`（Load-Exclusive）和 `STXR`（Store-Exclusive）指令对配合 `DMB`（Data Memory Barrier）内存屏障指令实现。无论哪种架构，最终都依赖**缓存一致性协议（MESI/MOESI）** 来保证多核 CPU 之间的数据一致性。

## 三、x86 与 ARM 的 CAS 实现差异

这是 Android 开发者特别需要关注的点——因为 Android 设备绝大多数是 ARM 架构，而我们熟悉的 Java 并发知识往往基于 x86 服务器环境。


**x86 的 CAS 实现（左侧）**：x86 采用**强内存模型（TSO: Total Store Order）**，处理器保证写操作对其他处理器的可见顺序与程序顺序一致。`LOCK CMPXCHG` 指令通过锁定缓存行（现代 CPU 优先）或总线（降级方案）来保证原子性。由于强内存模型的保证，x86 上的 CAS 天然具备较强的顺序性语义。

**ARM 的 CAS 实现（右侧）**：ARM 采用**弱内存模型**，处理器可能会对指令进行激进的重排序以提升性能。因此，ARM 的原子操作需要显式的内存屏障配合。`LDXR`（Load-Exclusive）指令加载内存值并标记该地址为"独占访问"；`STXR`（Store-Exclusive）指令尝试写入，只有当独占标记仍然有效时才能成功。如果在 `LDXR` 和 `STXR` 之间有其他核心访问了同一地址，独占标记会被清除，`STXR` 返回失败，需要重试。`DMB`（Data Memory Barrier）指令用于确保屏障前的所有内存访问完成后，屏障后的访问才开始。

**对 Android 开发的影响**：在纯 Java/Kotlin 层，ART 虚拟机已经为你处理了这些差异——当你调用 `AtomicInteger.compareAndSet()` 时，ART 会在 ARM 上自动插入必要的内存屏障。但如果你编写 NDK 代码直接操作原子变量，就必须显式使用 `<stdatomic.h>` 或 `__atomic_*` 内建函数，并指定正确的内存序（memory order），否则在 ARM 设备上可能出现诡异的并发 Bug。

## 四、Unsafe 类：CAS 的 Java 层入口

`sun.misc.Unsafe` 是 Java 并发包的"核反应堆"——极其危险但又不可或缺。所有原子类的 CAS 操作最终都委托给 `Unsafe`。

```java
// AtomicInteger 的核心实现（简化版）
public class AtomicInteger extends Number implements java.io.Serializable {
    // Unsafe 实例：通过反射或特权获取
    private static final Unsafe U = Unsafe.getUnsafe();
    
    // value 字段在对象内存中的偏移量（关键！）
    private static final long VALUE_OFFSET;
    
    static {
        try {
            // 获取 "value" 字段相对于对象起始地址的字节偏移
            VALUE_OFFSET = U.objectFieldOffset(
                AtomicInteger.class.getDeclaredField("value"));
        } catch (ReflectiveOperationException e) {
            throw new ExceptionInInitializerError(e);
        }
    }
    
    // volatile 保证可见性，CAS 保证原子性
    private volatile int value;
    
    // CAS 核心方法
    public final boolean compareAndSet(int expectedValue, int newValue) {
        // 本质：Unsafe.compareAndSwapInt(对象, 偏移量, 期望值, 新值)
        return U.compareAndSwapInt(this, VALUE_OFFSET, expectedValue, newValue);
    }
    
    // 自增并返回旧值（典型的 CAS 自旋模式）
    public final int getAndIncrement() {
        return U.getAndAddInt(this, VALUE_OFFSET, 1);
    }
}
```

**关键细节解析**：

1. **`VALUE_OFFSET` 的含义**：这是 `value` 字段在 `AtomicInteger` 对象内存布局中的字节偏移量。JVM/ART 的对象内存布局包括对象头（Mark Word + Class Pointer）和实例数据。`Unsafe.objectFieldOffset()` 计算出 `value` 字段相对于对象起始地址的偏移，后续 CAS 操作直接基于这个偏移进行内存级别的比较与交换。

2. **`volatile` 与 CAS 的配合**：`value` 被声明为 `volatile`，这保证了读取时的**可见性**（总是读取主存最新值）；CAS 保证了修改时的**原子性**。两者缺一不可——如果去掉 `volatile`，即使 CAS 成功，其他线程也可能读取到 CPU 缓存中的旧值。

3. **`Unsafe.compareAndSwapInt()` 的 native 实现**：这个方法最终会调用 ART 的 native 代码，在 ARM 上生成 `LDXR/STXR` 指令序列，在 x86 上生成 `LOCK CMPXCHG` 指令。

## 五、典型落地场景与避坑指南

### 场景 1：线程安全计数器（正确姿势）

```kotlin
// ✅ 正确：使用 AtomicInteger
class RequestCounter {
    private val count = AtomicInteger(0)
    
    fun onRequest() {
        val current = count.incrementAndGet()
        Log.d("Counter", "当前请求数: $current")
    }
    
    fun getCount(): Int = count.get()
}

// ❌ 错误：volatile 不保证原子性
class BrokenCounter {
    @Volatile
    private var count = 0
    
    fun onRequest() {
        count++  // 这不是原子操作！包含 读取-修改-写入 三步
    }
}
```

`count++` 在字节码层面是三条指令：`GETFIELD`（读取）、`IADD`（加一）、`PUTFIELD`（写回）。在两条指令之间，其他线程可能已经修改了 `count`，导致更新丢失。

### 场景 2：单例模式的 DCL 问题与 CAS 替代方案

```kotlin
// 传统 DCL（Double-Checked Locking）
class Singleton private constructor() {
    companion object {
        @Volatile
        private var instance: Singleton? = null
        
        fun getInstance(): Singleton {
            if (instance == null) {                    // 第一次检查
                synchronized(this) {
                    if (instance == null) {            // 第二次检查
                        instance = Singleton()
                    }
                }
            }
            return instance!!
        }
    }
}

// CAS 实现（无锁方案）
class CasSingleton private constructor() {
    companion object {
        private val INSTANCE = AtomicReference<CasSingleton?>(null)
        
        fun getInstance(): CasSingleton {
            // CAS 自旋直到成功设置或发现已被设置
            while (true) {
                val current = INSTANCE.get()
                if (current != null) return current
                
                val newInstance = CasSingleton()
                if (INSTANCE.compareAndSet(null, newInstance)) {
                    return newInstance
                }
                // CAS 失败说明其他线程抢先创建，下一轮循环会返回已创建的实例
            }
        }
    }
}
```

**CAS 方案的权衡**：CAS 单例可能会创建多个实例（虽然只有一个会被采用），在构造函数开销大时不划算。但它避免了 `synchronized` 的锁竞争，在对象创建开销小、并发度高的场景更优。

### 场景 3：ABA 问题——CAS 的阿喀琉斯之踵

```mermaid
sequenceDiagram
    autonumber
    participant T1 as 线程 T1
    participant Mem as 共享内存<br/>(初始值 A)
    participant T2 as 线程 T2

    T1->>Mem: 读取值 = A<br/>准备 CAS(A → C)
    Note right of T1: T1 被调度器挂起...
    
    T2->>Mem: CAS(A → B) 成功
    Note right of Mem: 值变为 B
    T2->>Mem: CAS(B → A) 成功
    Note right of Mem: 值变回 A<br/>(但语义已变！)
    
    T1->>Mem: 恢复执行<br/>CAS(A → C) 成功！
    Note right of T1: T1 认为值"没变过"<br/>实际上经历了 A→B→A
```

**ABA 问题的本质**：CAS 只比较"值"是否相等，无法感知值是否经历过中间状态的变化。在操作链表节点、内存池对象等场景下，ABA 可能导致严重的逻辑错误——一个被删除又重新分配的节点，其内存地址（即"值"）可能与之前相同，但节点内容已完全不同。

**解决方案：版本号/时间戳**

```java
// AtomicStampedReference：值 + 版本号
AtomicStampedReference<String> ref = new AtomicStampedReference<>("A", 0);

// CAS 时同时比较值和版本号
int[] stampHolder = new int[1];
String current = ref.get(stampHolder);  // 获取当前值和版本号
int currentStamp = stampHolder[0];

// 只有值和版本号都匹配才能成功
boolean success = ref.compareAndSet(
    current,           // 期望的值
    "C",               // 新值
    currentStamp,      // 期望的版本号
    currentStamp + 1   // 新版本号
);
```

`AtomicStampedReference` 内部将引用和版本号打包成一个 `Pair` 对象，对这个 `Pair` 进行 CAS 操作。这样即使值从 A→B→A，版本号也会从 0→1→2 单调递增，CAS 能检测到变化。

### 场景 4：CAS 自旋的性能陷阱——活锁（Livelock）

CAS 自旋在高竞争场景下可能导致**活锁**：所有线程都在"忙碌"地重试，但没有任何线程能取得实质性进展，CPU 空转但吞吐量趋近于零。

**图解分析**：当 4 个线程同时对同一个 `AtomicInteger` 执行 `incrementAndGet()` 时，它们都读取到相同的初始值 100。但 CAS 操作是互斥的——只有一个线程能成功将值从 100 改为 101，其他三个线程的 CAS 全部失败。失败的线程必须重新读取最新值（101），然后再次尝试 CAS(101→102)。在这一轮中，又只有一个线程能成功。**如果线程数远大于 CPU 核心数，大量线程会陷入"读取→CAS失败→重读→再CAS失败"的死循环**，CPU 利用率 100% 但有效吞吐量极低。

**性能量化对比**：

```java
// 基准测试场景：64 线程并发对计数器累加 100万次
// 测试设备：8 核 ARM64 (Snapdragon 888)

// 方案 1: AtomicLong
AtomicLong counter1 = new AtomicLong(0);
// 结果：~1200ms，CAS 冲突率 > 90%

// 方案 2: LongAdder（分段累加）
LongAdder counter2 = new LongAdder();
// 结果：~180ms，冲突率 < 5%

// 方案 3: synchronized
long counter3 = 0;
synchronized(lock) { counter3++; }
// 结果：~2800ms，大量线程阻塞与唤醒
```

**为什么 LongAdder 快 6 倍以上？** 这正是阶段 2 将要深入剖析的核心——**分段累加（Striped64）** 机制。

### 场景 5：Android Framework 中的 CAS 实战——以 `ViewRootImpl` 为例

CAS 并非仅存在于 `java.util.concurrent` 包中，它深入渗透到 Android Framework 的核心组件。以下是 `ViewRootImpl`（负责 View 树的绘制调度）中的一个典型应用：

```java
// frameworks/base/core/java/android/view/ViewRootImpl.java
public final class ViewRootImpl implements ViewParent {
    
    // 使用 AtomicInteger 管理绘制请求的序列号
    // 用于检测"过期的绘制回调"
    private final AtomicInteger mDrawsNeededToReport = new AtomicInteger(0);
    
    // 调度一次 Vsync 同步的绘制
    void scheduleTraversals() {
        if (!mTraversalScheduled) {
            mTraversalScheduled = true;
            // ...
            // 原子递增，记录"待处理的绘制请求数"
            mDrawsNeededToReport.incrementAndGet();
            // 通过 Choreographer 注册 Vsync 回调
            mChoreographer.postCallback(
                Choreographer.CALLBACK_TRAVERSAL, mTraversalRunnable, null);
        }
    }
    
    // Vsync 回调触发后执行
    void doTraversal() {
        if (mTraversalScheduled) {
            mTraversalScheduled = false;
            // 执行 measure → layout → draw
            performTraversals();
            // 原子递减
            int remaining = mDrawsNeededToReport.decrementAndGet();
            if (remaining == 0) {
                // 所有绘制请求都已完成，可以通知监听者
                reportDrawFinished();
            }
        }
    }
}
```

**为什么这里使用 AtomicInteger 而非 synchronized？**

1. **性能敏感路径**：`scheduleTraversals()` 和 `doTraversal()` 位于 UI 渲染的关键路径上，每一帧都可能调用。如果使用 `synchronized` 阻塞，可能导致掉帧。

2. **竞争窗口极小**：这两个方法虽然可能被不同线程调用（如主线程和 RenderThread），但实际冲突概率很低。CAS 的"乐观"特性在此场景下收益最大。

3. **语义清晰**：`incrementAndGet()` 和 `decrementAndGet()` 的原子性语义比 `synchronized { count++ }` 更直观。

**类族深度解读**：

**基本类型原子类（绿色，最常用）**：`AtomicInteger`、`AtomicLong`、`AtomicBoolean` 是最基础的原子类，内部通过 `Unsafe.compareAndSwapInt/Long` 实现 CAS。`AtomicBoolean` 实际上是将 `boolean` 映射为 `int`（0/1）进行 CAS 操作，因为 CPU 的 CAS 指令通常只支持 32 位或 64 位操作数。

**引用类型原子类（绿色）**：`AtomicReference<V>` 对对象引用进行 CAS，本质是对指针（在 JVM 中是 oop，Ordinary Object Pointer）的比较与交换。`AtomicStampedReference` 和 `AtomicMarkableReference` 是针对 **ABA 问题** 的解决方案——前者使用 `int` 类型的版本号，后者使用 `boolean` 类型的标记位。

**数组原子类（绿色）**：支持对数组中**特定索引位置**的元素进行原子操作。内部实现需要计算元素的内存偏移量：`base + index * scale`，其中 `base` 是数组第一个元素的偏移，`scale` 是每个元素的字节大小。

**字段更新器（蓝色，高级用法）**：用于对**已有类**中的 `volatile` 字段进行原子更新，无需修改原类的代码。这在无法直接修改源码（如系统类或第三方库类）但又需要原子操作时非常有用。缺点是依赖反射，有一定性能开销，且 Android 上对非公开 API 的反射有限制。

**累加器（蓝色，高并发首选）**：`LongAdder` 和 `DoubleAdder` 是 JDK 8 引入的高并发计数器优化方案。它们继承自 `Striped64`，采用**分段累加**策略——内部维护一个 `Cell[]` 数组，不同线程映射到不同的 Cell 进行累加，最后 `sum()` 时汇总所有 Cell 的值。这大幅降低了 CAS 冲突概率，是高并发统计场景的最佳实践。

**底层支撑（橙色）**：`Unsafe` 是 CAS 操作的终极入口；`Striped64` 是 `LongAdder` 家族的抽象基类，封装了分段策略和 Cell 数组的管理逻辑。

## 七、字段更新器的特殊场景：突破封装的原子操作

`AtomicIntegerFieldUpdater` 等字段更新器是一个容易被忽视但极具价值的工具。考虑以下场景：

```java
// 你无法修改的第三方库类
public class ThirdPartyNode {
    volatile int refCount;  // 引用计数，需要原子操作
    // ... 其他字段和方法
}

// 你的代码：使用字段更新器实现原子操作
public class NodeManager {
    // 创建更新器（静态常量，只创建一次）
    private static final AtomicIntegerFieldUpdater<ThirdPartyNode> REF_COUNT_UPDATER =
        AtomicIntegerFieldUpdater.newUpdater(ThirdPartyNode.class, "refCount");
    
    public void retain(ThirdPartyNode node) {
        REF_COUNT_UPDATER.incrementAndGet(node);  // 原子递增
    }
    
    public boolean release(ThirdPartyNode node) {
        int newCount = REF_COUNT_UPDATER.decrementAndGet(node);  // 原子递减
        if (newCount == 0) {
            // 引用计数归零，可以回收
            recycle(node);
            return true;
        }
        return false;
    }
}
```

**字段更新器的使用约束**：

1. 目标字段必须是 `volatile` 的
2. 目标字段不能是 `static` 的（static 字段用 `AtomicIntegerFieldUpdater` 无法更新）
3. 目标字段的可见性必须允许调用者访问（Android 的隐藏 API 限制在此生效）
4. 字段类型必须精确匹配（`int` 对应 `AtomicIntegerFieldUpdater`，不能用于 `Integer` 包装类型）

**Android 上的注意事项**：从 Android 9 开始，Google 对非 SDK 接口（灰名单/黑名单）进行了限制。如果你尝试对系统类的非公开字段使用字段更新器，可能会在运行时抛出异常或静默失败。在做这类"黑魔法"操作前，务必通过 `StrictMode` 或 `@hide` 注解检查目标字段的开放程度。

| 轻量级锁获取      | 50-200            | 用户态，CAS + 自旋 |
| 重量级锁获取      | 1000-10000+       | 涉及系统调用、上下文切换 |
| 线程唤醒        | 5000-20000+       | 调度器介入、缓存重建   |

**设计洞察**：对于**临界区极短**（如简单的计数器递增、状态标志位修改）的场景，`synchronized` 的"安全网"机制反而成为性能瓶颈——即使实际冲突率很低，每次进入临界区都要执行锁获取/释放的完整流程。CAS 的设计哲学是**"假设冲突不发生，发生时再处理"**，对于这类场景能节省大量开销。

## 九、AtomicInteger 源码逐行解剖

### 9.1 核心字段与静态初始化

```java
// java.util.concurrent.atomic.AtomicInteger (JDK 源码简化版)
public class AtomicInteger extends Number implements java.io.Serializable {
    private static final long serialVersionUID = 6214790243416807050L;

    /*
     * 核心依赖：Unsafe 实例
     * Unsafe 是 JDK 内部类，提供底层内存操作能力
     * 普通应用无法直接获取（会抛出 SecurityException）
     * 但 JDK 内部类可以通过 theUnsafe 字段访问
     */
    private static final jdk.internal.misc.Unsafe U = jdk.internal.misc.Unsafe.getUnsafe();
    
    /*
     * VALUE 字段在对象内存中的偏移量
     * 这是 CAS 操作的关键——直接操作内存地址
     * 
     * 为什么需要偏移量？
     * - Java 对象在内存中的布局：对象头(Header) + 实例数据(Instance Data) + 对齐填充(Padding)
     * - CAS 需要知道 value 字段相对于对象起始地址的精确位置
     */
    private static final long VALUE = U.objectFieldOffset(AtomicInteger.class, "value");

    /*
     * 实际存储的 int 值
     * volatile 保证：
     * 1. 可见性：写操作立即刷新到主存，读操作从主存读取
     * 2. 有序性：volatile 写之前的操作不会重排到写之后
     *            volatile 读之后的操作不会重排到读之前
     */
    private volatile int value;

    // 构造函数
    public AtomicInteger(int initialValue) {
        value = initialValue;
    }

    public AtomicInteger() {
        // 默认值 0（Java 语言规范保证）
    }
}
```

**关键细节——`VALUE` 偏移量的计算**：

```c
// 对象内存布局示意 (64位 JVM，压缩指针开启)
+---------------------------+
|   Mark Word (8 bytes)     |  // 对象头：锁状态、GC 年龄、哈希码
+---------------------------+
|   Class Pointer (4 bytes) |  // 类指针（压缩后）
+---------------------------+
|   value (4 bytes)         |  // <-- VALUE 偏移量指向这里 (offset = 12)
+---------------------------+
|   Padding (0 bytes)       |  // 对齐填充（本例无需）
+---------------------------+
```

`U.objectFieldOffset(AtomicInteger.class, "value")` 返回的就是 `12`（在 64 位压缩指针环境下）。后续 CAS 操作会直接在 `对象地址 + 12` 的位置进行原子比较与交换。

### 9.2 核心方法：compareAndSet 与 getAndIncrement

```java
public class AtomicInteger extends Number implements java.io.Serializable {
    
    /*
     * 最基础的 CAS 操作
     * 语义：如果当前值 == expectedValue，则更新为 newValue，返回 true
     *       否则不做任何修改，返回 false
     * 
     * 这是一个"原子弱 CAS"(weakCompareAndSet 已废弃，现在统一使用这个)
     */
    public final boolean compareAndSet(int expectedValue, int newValue) {
        return U.compareAndSetInt(this, VALUE, expectedValue, newValue);
    }

    /*
     * 原子递增，返回旧值（类似 i++）
     * 实现：CAS 自旋循环
     */
    public final int getAndIncrement() {
        return U.getAndAddInt(this, VALUE, 1);
    }

    /*
     * 原子递增，返回新值（类似 ++i）
     */
    public final int incrementAndGet() {
        return U.getAndAddInt(this, VALUE, 1) + 1;
    }

    /*
     * 原子更新：基于当前值计算新值
     * updateFunction: 接收当前值，返回新值的函数
     * 返回：更新后的值
     * 
     * 适用于复杂的更新逻辑，如：atomicInt.updateAndGet(x -> x * 2 + 1)
     */
    public final int updateAndGet(IntUnaryOperator updateFunction) {
        int prev = get(), next = 0;
        for (boolean haveNext = false;;) {
            if (!haveNext)
                next = updateFunction.applyAsInt(prev);  // 计算新值
            // CAS 尝试更新
            if (weakCompareAndSetVolatile(prev, next))
                return next;  // 成功，返回新值
            haveNext = (prev == (prev = get()));  // 失败，重读当前值
            // 如果值没变（prev == 新读取的 prev），可以复用 next，避免重复计算
        }
    }
}
```

**`getAndAddInt` 的 Unsafe 实现**：

```java
// jdk.internal.misc.Unsafe (JDK 内部实现)
public final class Unsafe {
    
    /*
     * 原子地将 delta 加到 offset 位置的 int 值上，返回旧值
     * 这是一个"自旋 CAS"的标准实现
     */
    @IntrinsicCandidate  // 提示 JIT 编译器：可以替换为 CPU 原生指令
    public final int getAndAddInt(Object o, long offset, int delta) {
        int v;
        do {
            // Step 1: volatile 读取当前值
            v = getIntVolatile(o, offset);
            // Step 2: CAS 尝试更新 (当前值 v -> 新值 v + delta)
            // 如果失败（说明其他线程修改了值），继续循环
        } while (!weakCompareAndSetInt(o, offset, v, v + delta));
        return v;  // 返回旧值
    }

    /*
     * 弱 CAS：可能伪失败（spurious failure）
     * 在某些架构上，即使值匹配，也可能返回 false（如 ARM 的 LL/SC 机制）
     * 但这不影响正确性，因为外层有 while 循环重试
     */
    @IntrinsicCandidate
    public final native boolean weakCompareAndSetInt(Object o, long offset,
                                                     int expected, int x);
}
```

**`@IntrinsicCandidate` 注解的意义**：这个注解告诉 JIT 编译器，该方法有对应的 **CPU 内联指令（Intrinsic）** 实现。编译时，JIT 不会生成常规的方法调用，而是直接嵌入对应的机器指令（如 x86 的 `LOCK CMPXCHG`），极大提升性能。

### 9.3 CAS 自旋的执行流程时序图

```mermaid
sequenceDiagram
    autonumber
    participant Thread as 业务线程
    participant Atomic as AtomicInteger
    participant Unsafe as Unsafe.getAndAddInt
    participant Memory as 主存 value

    Thread->>Atomic: incrementAndGet()
    Atomic->>Unsafe: getAndAddInt(this, VALUE, 1)
    
    loop CAS 自旋循环
        Unsafe->>Memory: getIntVolatile() 读取当前值
        Memory-->>Unsafe: 返回 v = 100
        Note right of Unsafe: 计算新值: v + 1 = 101
        Unsafe->>Memory: weakCompareAndSetInt(100, 101)
        alt CAS 成功
            Memory-->>Unsafe: true (值已更新为 101)
            Unsafe-->>Atomic: 返回旧值 100
        else CAS 失败 (其他线程已修改)
            Memory-->>Unsafe: false (当前值已不是 100)
            Note right of Unsafe: 继续下一轮循环
        end
    end
    
    Atomic-->>Thread: 返回新值 101 (旧值 + 1)
```

**时序图关键步骤解析**：

**步骤 1-3（读取当前值）**：`getIntVolatile()` 是一个 volatile 语义的读取，保证读取到的是主存中的最新值，而非 CPU 缓存中的旧值。在 ARM 架构上，这会在读取后插入一个 Load-Acquire 屏障。

**步骤 4（计算新值）**：这一步在用户态完成，不涉及任何原子操作或内存屏障。新值 `v + 1` 只存在于当前线程的局部变量中。

**步骤 5（CAS 尝试）**：`weakCompareAndSetInt` 是真正的原子操作。它在底层会生成 CPU 原子指令，将"比较"和"交换"作为一个不可分割的操作执行。如果当前内存值仍然是 100，则更新为 101 并返回 true；否则什么都不做，返回 false。

**步骤 6-7（成功/失败分支）**：CAS 成功时，整个操作完成；CAS 失败时，说明在步骤 1 和步骤 5 之间，有其他线程修改了值，当前线程的预期已经"过时"，需要重新读取最新值并再次尝试。这就是"乐观并发控制"的精髓——不预先阻塞，而是在提交时检测冲突。

## 十、CPU 指令层面：`CMPXCHG` 深度解析

### 10.1 x86 架构的 `LOCK CMPXCHG`

```asm
; x86-64 汇编：AtomicInteger.compareAndSet(100, 101) 的最终形态
; 假设对象地址在 RDI，VALUE 偏移量 = 12

mov     eax, 100                ; EAX = 期望值 (expected)
mov     ecx, 101                ; ECX = 新值 (desired)
lock cmpxchg [rdi+12], ecx      ; 原子比较并交换

; CMPXCHG 指令语义：
; 1. 比较 EAX 与 [RDI+12] 的值
; 2. 如果相等：[RDI+12] = ECX, ZF = 1 (成功)
; 3. 如果不等：EAX = [RDI+12], ZF = 0 (失败，EAX 被更新为实际值)

; LOCK 前缀的作用：
; - 锁定内存总线或缓存行，保证操作期间其他 CPU 无法访问该地址
; - 隐含 Full Memory Barrier（全内存屏障）效果
```

**`LOCK` 前缀的演进**：

| 时代     | 实现方式            | 性能影响                     |
| ------ | --------------- | ------------------------ |
| 古早 CPU | 总线锁（Bus Lock）   | 锁定整个内存总线，其他 CPU 完全停滞     |
| 现代 CPU | 缓存锁（Cache Lock） | 仅锁定目标缓存行，通过 MESI 协议保证一致性 |

现代 CPU 的 `LOCK CMPXCHG` 几乎不会锁定总线，而是利用**缓存一致性协议**：执行 CAS 的 CPU 将目标缓存行置为 Exclusive 状态，其他 CPU 的同一缓存行被置为 Invalid。只有当目标地址跨越两个缓存行（极端情况）时，才会退化为总线锁。

### 10.2 ARM 架构的 `LDXR/STXR`

ARM 采用与 x86 完全不同的原子操作模型：**Load-Link / Store-Conditional（LL/SC）**。

```asm
; ARM64 汇编：AtomicInteger.compareAndSet(100, 101)
; 假设对象地址在 X0，VALUE 偏移量 = 12

    add     x1, x0, #12         ; X1 = &value (对象地址 + 偏移量)
    mov     w2, #100            ; W2 = 期望值
    mov     w3, #101            ; W3 = 新值

retry:
    ldxr    w4, [x1]            ; 独占加载：W4 = *X1，并标记 X1 为"独占访问"
    cmp     w4, w2              ; 比较当前值与期望值
    b.ne    fail                ; 不相等，跳转到失败处理

    stxr    w5, w3, [x1]        ; 独占存储：尝试 *X1 = W3
                                 ; W5 = 0 表示成功，W5 = 1 表示失败
    cbnz    w5, retry           ; 如果 W5 != 0（存储失败），重试
    
    dmb     ish                 ; 数据内存屏障：确保后续操作看到最新值
    ; ... 成功路径 ...

fail:
    clrex                       ; 清除独占标记
    ; ... 失败路径 ...
```

**LDXR/STXR 的工作原理**：

**独占监视器（Exclusive Monitor）** 是 ARM 架构的核心机制。每个 CPU 核心都有一个本地独占监视器，当执行 `LDXR` 时，监视器会记录"该地址被本核心独占访问"。如果在 `LDXR` 和 `STXR` 之间，任何其他核心（或 DMA）访问了同一地址，独占标记会被清除，`STXR` 返回失败。

**与 x86 CMPXCHG 的关键差异**：

| 特性   | x86 LOCK CMPXCHG | ARM LDXR/STXR        |
| ---- | ---------------- | -------------------- |
| 失败条件 | 仅当值不匹配           | 值不匹配 **或** 地址被其他核心触碰 |
| 伪失败  | 不会               | 可能（spurious failure） |
| 内存屏障 | LOCK 隐含全屏障       | 需显式 DMB 指令           |
| 性能   | 较重（锁缓存行）         | 较轻（仅监视器检查）           |

**伪失败（Spurious Failure）的影响**：在 ARM 上，即使没有真正的数据竞争，`STXR` 也可能返回失败（如中断、上下文切换触发独占标记清除）。这就是为什么 Unsafe 提供 `weakCompareAndSet`（允许伪失败）和 `compareAndSet`（循环重试直到真正成功）两个版本。对于 `getAndAddInt` 这类本身就有外层循环的场景，使用 `weakCompareAndSet` 更高效。

## 十一、LongAdder 深度剖析：分段累加的艺术

`LongAdder` 是 JDK 8 引入的高并发计数器，专门解决 `AtomicLong` 在高竞争场景下的性能瓶颈。其核心思想是**"分散热点，最终汇总"**——将单一的 CAS 热点分散到多个 Cell 上，降低冲突概率。

### 11.1 Striped64：LongAdder 的骨架

`LongAdder` 继承自 `Striped64`，后者是所有分段累加器的抽象基类。

```java
// java.util.concurrent.atomic.Striped64 (简化版)
abstract class Striped64 extends Number {
    
    /*
     * Cell 数组：分段累加的核心数据结构
     * 
     * 设计要点：
     * 1. 懒初始化：只有在发生竞争时才创建
     * 2. 长度必须是 2 的幂：方便通过位运算定位 (hash & (n-1))
     * 3. 只扩容不缩容：避免数据丢失
     */
    transient volatile Cell[] cells;

    /*
     * 基础值：无竞争时直接在 base 上累加
     * 
     * 优化思路：
     * - 单线程或低竞争时，直接 CAS base，避免 Cell 数组开销
     * - 只有 CAS base 失败时，才会尝试使用 Cell
     */
    transient volatile long base;

    /*
     * 自旋锁：用于 Cell 数组的初始化和扩容
     * 0 = 未锁定, 1 = 已锁定
     * 
     * 为什么不用 ReentrantLock？
     * - 初始化/扩容是极低频操作，不值得引入重量级锁
     * - CAS 自旋足够高效，且避免了锁的额外内存开销
     */
    transient volatile int cellsBusy;

    /*
     * Cell 内部类：每个 Cell 持有一个独立的累加值
     * 
     * @Contended 注解的作用：
     * - 要求 JVM 在 Cell 对象前后填充 padding，避免伪共享
     * - 伪共享：多个 Cell 落在同一个缓存行，一个 Cell 的更新导致
     *   整个缓存行失效，其他 Cell 的读取也被迫从主存重载
     */
    @jdk.internal.vm.annotation.Contended
    static final class Cell {
        volatile long value;
        
        Cell(long x) { value = x; }
        
        // Cell 内部也使用 CAS 更新
        final boolean cas(long cmp, long val) {
            return VALUE.compareAndSet(this, cmp, val);
        }
        
        private static final VarHandle VALUE;
        static {
            try {
                VALUE = MethodHandles.lookup()
                    .findVarHandle(Cell.class, "value", long.class);
            } catch (ReflectiveOperationException e) {
                throw new ExceptionInInitializerError(e);
            }
        }
    }
}
```

**`@Contended` 与伪共享问题**：

```c
// 缓存行布局对比 (假设缓存行大小 64 字节)

// ❌ 无 @Contended：多个 Cell 可能落在同一缓存行
+------------------------------------------------------------------+
|  Cell[0].value (8B) | Cell[1].value (8B) | Cell[2].value (8B) ...|
+------------------------------------------------------------------+
                        ^
                        |
                  一个 Cell 更新，整行失效！

// ✅ 有 @Contended：每个 Cell 独占缓存行
+------------------------------------------------------------------+
|  padding (128B) | Cell[0].value (8B) | padding (128B)            |
+------------------------------------------------------------------+
+------------------------------------------------------------------+
|  padding (128B) | Cell[1].value (8B) | padding (128B)            |
+------------------------------------------------------------------+
                        ^
                        |
                  更新互不影响
```

**伪共享（False Sharing）** 是多核 CPU 并发编程中的隐蔽性能杀手。CPU 缓存以"缓存行"（通常 64 字节）为单位管理数据。当两个线程分别修改同一缓存行中的不同变量时，每次修改都会导致另一个核心的缓存行失效，即使两个变量在逻辑上完全独立。`@Contended` 通过在对象前后插入 128 字节的 padding，确保每个 Cell 独占至少一个缓存行。

**流程深度解析**：

**快速路径（绿色→蓝色）**：这是 LongAdder 的第一道优化。当系统处于低竞争状态时，所有累加操作直接 CAS 到 `base` 字段上，完全不涉及 Cell 数组。这使得 LongAdder 在低竞争场景下的性能与 AtomicLong 相当，不会因为"分段"机制引入额外开销。

**慢速路径入口**：当首次 CAS `base` 失败（说明有竞争），或者 `cells` 数组已存在时，进入慢速路径。此时需要将线程"分流"到不同的 Cell 上。

**线程 hash 计算**：`getProbe()` 返回当前线程的"探针值"（probe），这是一个存储在 `Thread` 对象中的伪随机数，用于将线程映射到 Cell 数组的某个槽位。与 `hashCode()` 不同，probe 值会在 CAS 失败时**重新计算**，实现"冲突后换个位置试试"的效果。

**Cell 定位与更新**：通过 `hash & (cells.length - 1)` 定位到目标 Cell（位运算取模，要求数组长度为 2 的幂）。如果目标 Cell 存在且 CAS 成功，操作完成；否则进入 `longAccumulate()` 进行更复杂的处理。

### 11.3 longAccumulate()：初始化与扩容的核心

```java
// Striped64.longAccumulate() 简化版
final void longAccumulate(long x, LongBinaryOperator fn, boolean wasUncontended) {
    int h;  // 当前线程的 probe 值
    if ((h = getProbe()) == 0) {
        // probe 未初始化（新线程），强制初始化
        ThreadLocalRandom.current();
        h = getProbe();
        wasUncontended = true;  // 重新计算 hash 后，视为无竞争状态
    }

    boolean collide = false;  // 是否发生过"真正的冲突"（同一个 Cell 上的 CAS 失败）

    for (;;) {  // 无限循环，直到成功
        Cell[] cs; Cell c; int n; long v;
        
        // === 分支 1：cells 数组已存在 ===
        if ((cs = cells) != null && (n = cs.length) > 0) {
            
            // 情况 1a：目标槽位为空，尝试创建新 Cell
            if ((c = cs[(n - 1) & h]) == null) {
                if (cellsBusy == 0) {       // 检查自旋锁
                    Cell r = new Cell(x);   // 预创建 Cell
                    if (cellsBusy == 0 && casCellsBusy()) {  // 获取锁
                        try {
                            // 双重检查：锁内再次确认槽位为空
                            Cell[] rs; int m, j;
                            if ((rs = cells) != null && (m = rs.length) > 0 &&
                                rs[j = (m - 1) & h] == null) {
                                rs[j] = r;  // 安装新 Cell
                                break;      // 成功，退出循环
                            }
                        } finally {
                            cellsBusy = 0;  // 释放锁
                        }
                        continue;  // 槽位被其他线程占用，重试
                    }
                }
                collide = false;  // 有锁竞争但不是 Cell CAS 冲突
            }
            
            // 情况 1b：上一次 CAS 失败（wasUncontended=false），重新计算 hash
            else if (!wasUncontended) {
                wasUncontended = true;
            }
            
            // 情况 1c：尝试 CAS 更新目标 Cell
            else if (c.cas(v = c.value, (fn == null) ? v + x : fn.applyAsLong(v, x))) {
                break;  // 成功，退出循环
            }
            
            // 情况 1d：Cell 数组已达到 CPU 核心数上限，或数组已被其他线程扩容
            else if (n >= NCPU || cells != cs) {
                collide = false;  // 不再扩容，只重新 hash
            }
            
            // 情况 1e：发生真正的冲突，标记 collide=true，下一轮尝试扩容
            else if (!collide) {
                collide = true;
            }
            
            // 情况 1f：尝试扩容
            else if (cellsBusy == 0 && casCellsBusy()) {
                try {
                    if (cells == cs) {  // 双重检查
                        cells = Arrays.copyOf(cs, n << 1);  // 扩容为 2 倍
                    }
                } finally {
                    cellsBusy = 0;
                }
                collide = false;
                continue;  // 扩容后重试
            }
            
            // 重新计算 probe，尝试换个槽位
            h = advanceProbe(h);
        }
        
        // === 分支 2：cells 数组尚未创建，尝试初始化 ===
        else if (cellsBusy == 0 && cells == cs && casCellsBusy()) {
            try {
                if (cells == cs) {  // 双重检查
                    Cell[] rs = new Cell[2];  // 初始大小为 2
                    rs[h & 1] = new Cell(x);  // 在对应槽位创建 Cell
                    cells = rs;
                    break;  // 成功，退出循环
                }
            } finally {
                cellsBusy = 0;
            }
        }
        
        // === 分支 3：初始化/扩容正在进行，退回 CAS base ===
        else if (casBase(v = base, (fn == null) ? v + x : fn.applyAsLong(v, x))) {
            break;  // 成功，退出循环
        }
    }
}
```

**扩容策略的精妙之处**：

1. **渐进式扩容**：不会一次性扩容到最大，而是从 2 开始，每次翻倍，最大不超过 CPU 核心数（`NCPU`）。这是因为 Cell 数组的理想大小等于并发线程数，而实际并发线程数不会超过 CPU 核心数。

2. **双重检查锁**：获取 `cellsBusy` 锁后，再次检查条件是否仍然满足，避免 ABA 式的竞态条件。

3. **冲突升级机制**：首次 CAS 失败时只是重新 hash（换个槽位）；连续失败才会触发扩容。这避免了因偶发冲突导致的过度扩容。

### 11.4 LongAdder.sum()：最终一致性汇总

```java
// LongAdder.sum()
public long sum() {
    Cell[] cs = cells;
    long sum = base;  // 从 base 开始
    if (cs != null) {
        for (Cell c : cs) {
            if (c != null) {
                sum += c.value;  // 累加每个 Cell 的值
            }
        }
    }
    return sum;
}
```

**关键特性——弱一致性**：`sum()` 不加锁，遍历过程中其他线程可能正在修改 Cell 或 base。这意味着返回的值是**某一时刻的近似值**，而非精确的"快照"。对于统计计数、性能指标采集等场景，这种弱一致性是可接受的；但对于需要精确一致性的场景（如账户余额），应使用 `AtomicLong`。

## 十二、ABA 问题的深度剖析

### 12.1 ABA 问题为什么是"问题"？

在前面的场景分析中，我们已经看到 ABA 问题的基本形态。但要真正理解其危害，需要考察它在**链表操作**中的致命影响。

```mermaid
sequenceDiagram
    autonumber
    participant T1 as 线程 T1<br/>(执行 pop)
    participant Stack as 无锁栈
    participant T2 as 线程 T2<br/>(干扰者)

    Note over Stack: 初始状态: top → A → B → C

    T1->>Stack: 读取 top = A<br/>读取 A.next = B<br/>准备 CAS(A → B)
    Note right of T1: T1 被挂起...

    T2->>Stack: pop() 弹出 A
    Note over Stack: top → B → C<br/>A 被回收到对象池

    T2->>Stack: pop() 弹出 B
    Note over Stack: top → C

    T2->>Stack: 从对象池取出 A<br/>push(A)
    Note over Stack: top → A → C<br/>注意: A.next = C, 不再是 B!

    T1->>Stack: 恢复执行<br/>CAS(top, A, B) 成功!
    Note over Stack: top → B → ???<br/>B 已被弹出, 指向垃圾内存!

    Note right of T1: 灾难: 栈结构被破坏<br/>后续操作可能 Crash
```

**时序图深度解析**：

**步骤 1-2（T1 准备操作）**：线程 T1 执行 `pop()` 操作，读取到栈顶指针 `top = A`，并预读了 `A.next = B`（准备将 B 设为新栈顶）。此时 T1 被调度器挂起。

**步骤 3-5（T2 的干扰操作）**：线程 T2 连续执行两次 `pop()`，将 A 和 B 都弹出。A 被回收到对象池（内存池、对象复用池）。然后 T2 执行 `push(A)`——从对象池取出 A，并将其压入栈顶。**关键点**：此时 A 虽然地址不变，但其 `next` 指针已变为 C（而非原来的 B）。

**步骤 6-7（灾难发生）**：T1 恢复执行，执行 `CAS(top, A, B)`。由于栈顶仍然是 A，CAS 成功，`top` 被设为 B。但问题是——**B 已经被弹出，它不在栈中！** B 的 `next` 指针可能指向已释放的内存、其他数据结构，甚至是随机值。后续任何访问 `top.next` 的操作都会导致不可预测的行为。

**根本原因**：CAS 只比较指针的"值"（即内存地址），无法感知指针指向的对象是否经历过"删除→重用"的生命周期变化。

### 12.2 AtomicStampedReference 源码解析

```java
// java.util.concurrent.atomic.AtomicStampedReference
public class AtomicStampedReference<V> {

    /*
     * 核心设计：将"引用"和"版本号"打包成一个 Pair 对象
     * CAS 操作针对整个 Pair，而非单独的引用
     */
    private static class Pair<T> {
        final T reference;      // 实际引用
        final int stamp;        // 版本号/时间戳
        
        private Pair(T reference, int stamp) {
            this.reference = reference;
            this.stamp = stamp;
        }
        
        static <T> Pair<T> of(T reference, int stamp) {
            return new Pair<T>(reference, stamp);
        }
    }

    private volatile Pair<V> pair;

    public AtomicStampedReference(V initialRef, int initialStamp) {
        pair = Pair.of(initialRef, initialStamp);
    }

    /*
     * 获取当前引用
     * stampHolder[0] 会被设置为当前版本号
     * 
     * 为什么用数组而非返回 Pair？
     * - Java 不支持多返回值
     * - 数组是最轻量的"out 参数"模拟方式
     */
    public V get(int[] stampHolder) {
        Pair<V> pair = this.pair;
        stampHolder[0] = pair.stamp;
        return pair.reference;
    }

    /*
     * 核心 CAS 方法
     * 只有当"当前引用 == expectedReference"且"当前版本号 == expectedStamp"时，
     * 才更新为 newReference 和 newStamp
     */
    public boolean compareAndSet(V expectedReference, V newReference,
                                 int expectedStamp, int newStamp) {
        Pair<V> current = pair;
        return
            expectedReference == current.reference &&  // 引用匹配
            expectedStamp == current.stamp &&          // 版本号匹配
            ((newReference == current.reference &&     // 新值与旧值相同
              newStamp == current.stamp) ||            // 且版本号也相同（无需更新）
             casPair(current, Pair.of(newReference, newStamp)));  // 否则 CAS 更新 Pair
    }

    private boolean casPair(Pair<V> cmp, Pair<V> val) {
        return PAIR.compareAndSet(this, cmp, val);
    }

    private static final VarHandle PAIR;
    static {
        try {
            PAIR = MethodHandles.lookup().findVarHandle(
                AtomicStampedReference.class, "pair", Pair.class);
        } catch (ReflectiveOperationException e) {
            throw new ExceptionInInitializerError(e);
        }
    }
}
```

**设计精髓——Pair 的不可变性**：

```c
// Pair 对象的内存布局
+-----------------------------------+
|  Pair 对象头 (12 bytes)           |
+-----------------------------------+
|  reference (4 bytes, 压缩指针)    |  // final, 不可变
+-----------------------------------+
|  stamp (4 bytes, int)             |  // final, 不可变
+-----------------------------------+
```

`Pair` 的两个字段都是 `final` 的，这意味着每次更新都需要创建新的 `Pair` 对象。这看似低效，但带来了关键优势：

1. **原子性保证**：CAS 操作针对 `pair` 引用（一个 8 字节指针），而非 `reference` 和 `stamp` 两个独立字段。这避免了"引用已更新但版本号还没更新"的中间状态。

2. **无锁安全**：由于 `Pair` 不可变，读取操作无需任何同步——读到的 `Pair` 对象在整个生命周期内都是一致的。

**版本号溢出问题**：`stamp` 是 `int` 类型，最大值约 21 亿。对于长时间运行的系统，如果 CAS 频率极高（如每秒百万次），几十分钟就会溢出。溢出后 `stamp` 会变为负数并继续递增，**在数学上仍然是单调的**（考虑环绕），但如果业务逻辑依赖 `stamp > 某阈值` 的判断，可能出问题。实践中，可以在版本号达到阈值时重置，或使用 `AtomicStampedReference<Long>` 配合 `LongAdder` 作为版本号生成器。

### 12.3 AtomicMarkableReference：轻量级 ABA 解决方案

当只需要区分"是否发生过变化"而非"变化了几次"时，可以使用 `AtomicMarkableReference`。

```java
// 使用场景：标记节点是否被"逻辑删除"
public class ConcurrentLinkedList<E> {
    
    static class Node<E> {
        final E item;
        // 使用 AtomicMarkableReference 存储 next 指针和"已删除"标记
        final AtomicMarkableReference<Node<E>> next;
        
        Node(E item, Node<E> next) {
            this.item = item;
            this.next = new AtomicMarkableReference<>(next, false);
        }
    }
    
    /*
     * 两阶段删除：
     * 1. 逻辑删除：将 next 的 mark 设为 true（标记节点已删除）
     * 2. 物理删除：从链表中移除节点
     * 
     * 这种模式避免了 ABA 问题：
     * - 其他线程看到 mark=true，就知道该节点正在被删除
     * - 不会将"正在删除的节点"误认为"新插入的节点"
     */
    public boolean remove(E item) {
        Node<E> pred = head;
        Node<E> curr = pred.next.getReference();
        
        while (curr != null) {
            Node<E> succ = curr.next.getReference();
            if (curr.item.equals(item)) {
                // 阶段 1：逻辑删除（标记）
                if (!curr.next.compareAndSet(succ, succ, false, true)) {
                    continue;  // 标记失败，重试
                }
                // 阶段 2：物理删除（摘链）
                pred.next.compareAndSet(curr, succ, false, false);
                return true;
            }
            pred = curr;
            curr = succ;
        }
        return false;
    }
}
```

**与 AtomicStampedReference 的对比**：

| 特性   | AtomicStampedReference | AtomicMarkableReference |
| ---- | ---------------------- | ----------------------- |
| 标记类型 | `int` 版本号              | `boolean` 标记位           |
| 信息量  | 可记录精确变化次数              | 只能记录"是否变化过"             |
| 内存开销 | Pair 对象 + int          | Pair 对象 + boolean       |
| 适用场景 | 需要版本历史的复杂场景            | 只需"脏标记"的简单场景            |

### 14.1 Hook Unsafe 的攻击面

```java
    }
```

### 14.2 CAS 相关的 Crash/ANR 归因

#### 14.2.1 无限自旋导致的 ANR

```kotlin
// 危险代码示例：自定义 CAS 循环缺乏退出条件
class DangerousCounter {
    private val value = AtomicInteger(0)
    
    // ❌ 错误：无限自旋，可能导致 ANR
    fun dangerousIncrement() {
        while (true) {
            val current = value.get()
            // 假设这里有复杂的计算逻辑
            val next = complexCalculation(current)
            if (value.compareAndSet(current, next)) {
                break
            }
            // 没有任何退避策略，极端情况下永远无法成功
        }
    }
    
    private fun complexCalculation(input: Int): Int {
        // 耗时计算...
        Thread.sleep(10)  // 模拟耗时
        return input + 1
    }
}
```

**问题根源**：`complexCalculation()` 耗时较长，在计算期间其他线程可能已经多次修改 `value`，导致 CAS 持续失败。如果在主线程执行，会导致 ANR。

**正确实践**：

```kotlin
class SafeCounter {
    private val value = AtomicInteger(0)
    
    // ✅ 正确：限制重试次数 + 指数退避
    fun safeIncrement(): Boolean {
        var retries = 0
        val maxRetries = 10
        
        while (retries < maxRetries) {
            val current = value.get()
            val next = current + 1  // 保持计算简单
            
            if (value.compareAndSet(current, next)) {
                return true
            }
            
            retries++
            // 指数退避：避免所有线程同时重试
            if (retries > 3) {
                Thread.sleep((1L shl (retries - 3)).coerceAtMost(100))
            }
        }
        
        // 重试耗尽，降级为锁
        synchronized(this) {
            value.incrementAndGet()
        }
        return true
    }
}
```

#### 14.2.2 内存可见性导致的诡异 Bug

```java
// 一个隐蔽的 Bug：混用 Unsafe 和普通读写
public class VisibilityBug {
    private int normalField = 0;
    private AtomicInteger atomicField = new AtomicInteger(0);
    
    // 线程 A
    public void writer() {
        normalField = 42;                    // 普通写
        atomicField.set(1);                  // volatile 写
    }
    
    // 线程 B
    public void reader() {
        while (atomicField.get() == 0) {
            // 自旋等待
        }
        // 问题：normalField 一定是 42 吗？
        System.out.println(normalField);

```

**分析**：这段代码在 x86 上通常能正确工作（强内存模型），但在 ARM 上可能出问题。虽然 `atomicField.set(1)` 是 volatile 写，会在之前插入 StoreStore 屏障，但 `normalField = 42` 的可见性取决于编译器和 CPU 是否对其进行了重排。

```java
        .findVarHandle(VisibilityBug.class, "normalField", int.class);

    NORMAL_FIELD.setRelease(this, 42);  // Release 语义：之前的写都可见
    atomicField.set(1);
}

public void reader() {
    while (atomicField.get() == 0) {}
    int value = (int) NORMAL_FIELD.getAcquire(this);  // Acquire 语义
    System.out.println(value);  // 保证是 42
}
```

## 十五、Android Framework 中的高级应用

### 15.1 HandlerThread 的线程安全退出

```java
// frameworks/base/core/java/android/os/HandlerThread.java
public class HandlerThread extends Thread {
    // 使用 volatile + 普通赋值，而非 AtomicReference
    // 为什么？因为 mLooper 只会被赋值一次，不需要 CAS
    volatile Looper mLooper;
    private @Nullable Handler mHandler;

    @Override
    public void run() {
        mTid = Process.myTid();
        Looper.prepare();
        synchronized (this) {
            mLooper = Looper.myLooper();
            notifyAll();  // 唤醒等待 getLooper() 的线程
        }
        Process.setThreadPriority(mPriority);
        onLooperPrepared();
        Looper.loop();
        mTid = -1;
    }

    // 线程安全地获取 Looper
    public Looper getLooper() {
        if (!isAlive()) {
            return null;
        }
        
        boolean wasInterrupted = false;
        synchronized (this) {
            // 自旋等待 mLooper 被初始化
            while (isAlive() && mLooper == null) {
                try {
                    wait();  // 释放锁，等待 notifyAll
                } catch (InterruptedException e) {
                    wasInterrupted = true;
                }
            }
        }
        
        if (wasInterrupted) {
            Thread.currentThread().interrupt();
        }
        
        return mLooper;
    }

    // 安全退出：使用 AtomicBoolean 或类似机制标记退出状态
    public boolean quitSafely() {
        Looper looper = getLooper();
        if (looper != null) {
            looper.quitSafely();
            return true;
        }
        return false;
    }
}
```

**设计洞察**：`HandlerThread` 没有使用 `AtomicReference<Looper>`，而是使用 `volatile` + `synchronized` + `wait/notifyAll` 的组合。这是因为 `mLooper` 的赋值只发生一次，不需要 CAS 的"重试"语义，而 `wait/notifyAll` 更适合"等待条件满足"的场景。**选择正确的同步原语比一味追求"无锁"更重要**。

### 15.2 Choreographer 的帧率统计

```java
// frameworks/base/core/java/android/view/Choreographer.java
public final class Choreographer {
    
    // 使用原子变量统计跳帧
    private static final AtomicLong sFrameCount = new AtomicLong(0);
    private static final AtomicLong sSkippedFrames = new AtomicLong(0);
    
    void doFrame(long frameTimeNanos, int frame) {
        // 计算跳帧数
        final long jitterNanos = frameTimeNanos - mLastFrameTimeNanos;
        if (jitterNanos >= mFrameIntervalNanos) {
            final long skippedFrames = jitterNanos / mFrameIntervalNanos;
            if (skippedFrames >= SKIPPED_FRAME_WARNING_LIMIT) {
                Log.i(TAG, "Skipped " + skippedFrames + " frames!");
            }
            // 原子累加跳帧统计
            sSkippedFrames.addAndGet(skippedFrames);
        }
        sFrameCount.incrementAndGet();
        
        // ... 执行回调
    }
    
    // 供 Perfetto/Systrace 读取
    public static long getTotalFrameCount() {
        return sFrameCount.get();
    }
    
    public static long getSkippedFrameCount() {
        return sSkippedFrames.get();
    }
}
```

**为什么 Choreographer 使用 AtomicLong 而非 LongAdder？**

1. **读取频率高**：性能监控工具可能频繁读取帧率统计，`LongAdder.sum()` 需要遍历所有 Cell，开销较大。

2. **写入频率低**：每帧（16ms）只递增一次，远未达到 CAS 冲突的临界点。

3. **精确性需求**：帧率监控需要相对精确的数值，`AtomicLong` 的强一致性语义更合适。

**VarHandle 的五种访问模式**：

```java
// VarHandle 提供了精细化的内存访问语义控制
public class VarHandleAccessModes {
    
    private int value;
    private static final VarHandle VALUE;
    
    static {
        try {
            VALUE = MethodHandles.lookup()
                .findVarHandle(VarHandleAccessModes.class, "value", int.class);
        } catch (Exception e) {
            throw new ExceptionInInitializerError(e);
        }
    }
    
    // 1. Plain 模式：无任何内存屏障，等同于普通字段访问
    //    适用于：单线程场景，或配合外部同步
    public int getPlain() {
        return (int) VALUE.get(this);
    }
    
    // 2. Opaque 模式：保证原子性，但不保证顺序性
    //    适用于：只需要原子读写，不关心与其他变量的顺序
    public int getOpaque() {
        return (int) VALUE.getOpaque(this);
    }
    
    // 3. Acquire 模式：读操作后插入 LoadLoad + LoadStore 屏障
    //    语义：保证该读之后的所有读写，不会被重排到该读之前
    public int getAcquire() {
        return (int) VALUE.getAcquire(this);
    }
    
    // 4. Release 模式：写操作前插入 LoadStore + StoreStore 屏障
    //    语义：保证该写之前的所有读写，不会被重排到该写之后
    public void setRelease(int newValue) {
        VALUE.setRelease(this, newValue);
    }
    
    // 5. Volatile 模式：等同于 volatile 读写
    //    = Acquire 读 + Release 写 + 全局顺序一致性
    public int getVolatile() {
        return (int) VALUE.getVolatile(this);
    }
    
    public void setVolatile(int newValue) {
        VALUE.setVolatile(this, newValue);
    }
    
    // CAS 操作也支持不同的访问模式
    public boolean casVolatile(int expected, int newValue) {
        return VALUE.compareAndSet(this, expected, newValue);
    }
    
    public boolean casRelease(int expected, int newValue) {
        return VALUE.compareAndExchangeRelease(this, expected, newValue) == expected;
    }
}
```

**访问模式的性能差异**：

```c
// ARM64 上不同访问模式的指令生成对比

// Plain 读：无额外指令
ldr     w0, [x1, #OFFSET]

// Opaque 读：与 Plain 相同（ARM64 的基本加载已是原子的）
ldr     w0, [x1, #OFFSET]

// Acquire 读：加载后插入屏障

// Volatile 读：最强屏障

// Release 写：存储前插入屏障

// Volatile 写：存储前后都有屏障
dmb     ish
str     w0, [x1, #OFFSET]
dmb     ish
```

**设计洞察**：VarHandle 的五种访问模式对应了 Java 内存模型（JMM）中不同强度的内存序保证。在大多数业务场景中，Acquire/Release 语义已经足够，比 Volatile 语义更轻量。例如，在"生产者-消费者"模式中，生产者使用 Release 写，消费者使用 Acquire 读，即可保证数据的正确传递，无需使用开销更大的 Volatile 语义。

### 17.2 协程挂起点与原子操作的交互

```kotlin
// 危险示例：在协程中错误使用原子操作
class CoroutineAtomicPitfall {
    private val counter = AtomicInteger(0)
    
    // ❌ 错误：CAS 循环中包含挂起点
    suspend fun dangerousIncrement() {
        while (true) {
            val current = counter.get()
            
            // 假设这里有异步计算
            val next = computeNextValue(current)  // 挂起点！
            
            // 问题：挂起期间，其他协程可能已经修改了 counter
            // 但更严重的是：恢复时可能在不同的线程上执行
            if (counter.compareAndSet(current, next)) {
                break
            }
        }
    }
    
    private suspend fun computeNextValue(current: Int): Int {
        delay(10)  // 模拟异步计算
        return current + 1
    }
}
```

**问题分析**：

```mermaid
sequenceDiagram
    autonumber
    participant C1 as 协程 1<br/>(线程 A)
    participant D as Dispatcher
    participant C1R as 协程 1<br/>(线程 B, 恢复后)
    participant Counter as AtomicInteger

    C1->>Counter: get() 返回 100
    C1->>D: delay(10) 挂起
    Note right of C1: 协程 1 被挂起<br/>释放线程 A

    D->>C1R: 10ms 后恢复<br/>可能在线程 B 上
    Note right of C1R: current 仍是 100<br/>但 counter 可能已变

    alt 如果其他协程已修改
        Counter-->>C1R: false (当前值已不是 100)

    end
```

**正确实践**：

```kotlin
class CoroutineAtomicBestPractice {
    private val counter = AtomicInteger(0)
    
    // ✅ 正确：使用 Mutex 保护复杂的异步更新
    private val mutex = Mutex()
    
    suspend fun safeAsyncIncrement(): Int {
        return mutex.withLock {
            val current = counter.get()
            val next = computeNextValue(current)
            counter.set(next)  // 在锁内直接 set，无需 CAS
            next
        }
    }
    
    // ✅ 正确：对于简单的原子操作，不使用挂起函数
    fun simpleIncrement(): Int {
        return counter.incrementAndGet()
    }
    
    // ✅ 正确：使用 updateAndGet 封装 CAS 循环
    fun updateWithFunction(): Int {
        return counter.updateAndGet { current ->
            // 注意：这个 lambda 必须是纯函数，不能有副作用
            current * 2 + 1
        }
    }
}
```

**核心原则**：原子操作的 CAS 循环必须是**无挂起的纯计算**。如果更新逻辑需要异步操作（网络请求、数据库查询等），应使用 `Mutex` 或其他协程同步原语，而非在 CAS 循环中插入挂起点。

```c
template<bool kTransactionActive, bool kCheckTransaction>
inline bool Object::CasFieldStrongSequentiallyConsistentObject(
    MemberOffset field_offset,
    ObjPtr<Object> old_value,
    ObjPtr<Object> new_value) {
  // 1. 执行 CAS
  bool success = CasFieldStrongSequentiallyConsistentObjectWithoutWriteBarrier(
      field_offset, old_value, new_value);
```

**Q：能否用 CAS 实现一个简单的自旋锁？**

```java
public class CASSpinLock {
    private final AtomicInteger state = new AtomicInteger(0);
    // 0 = 未锁定, 1 = 已锁定
    
    private static final int UNLOCKED = 0;
    private static final int LOCKED = 1;
    
    public void lock() {
        // 自旋直到获取锁
        while (!state.compareAndSet(UNLOCKED, LOCKED)) {
            // 可选：Thread.onSpinWait() 提示 CPU 正在自旋
            // JDK 9+，在超线程 CPU 上可降低对兄弟线程的干扰
            Thread.onSpinWait();
        }
    }
    
    public void unlock() {
        // 直接设置为未锁定（无需 CAS，因为只有持锁线程会调用）
        state.set(UNLOCKED);
    }
    
    // 更安全的版本：检查是否是当前线程持有锁
    public boolean tryLock() {
        return state.compareAndSet(UNLOCKED, LOCKED);
    }
}
```

**注意**：这个简单实现存在几个问题：

1. **无公平性**：不保证等待最久的线程先获取锁
2. **无重入**：同一线程不能重复获取锁
3. **无超时**：可能永久自旋

生产环境应使用 `ReentrantLock` 或更完善的自旋锁实现。

**Q：AtomicLong 在 32 位系统上是如何保证原子性的？**

```java
// 32 位 JVM 上，long 是 64 位，无法用单条 CAS 指令完成
// 解决方案：使用 synchronized 降级

// Unsafe.compareAndSwapLong 在 32 位系统上的实现（简化）
public final native boolean compareAndSwapLong(Object o, long offset,
                                               long expected, long x);

// JVM 内部实现（伪代码）
#ifdef __LP64__  // 64 位系统
    // 直接使用 64 位 CAS 指令
    return __sync_bool_compare_and_swap((volatile long*)(addr + offset), 
                                        expected, x);
#else  // 32 位系统
    // 需要锁保护
    pthread_mutex_lock(&global_long_cas_lock);
    long current = *(volatile long*)(addr + offset);
    if (current == expected) {
        *(volatile long*)(addr + offset) = x;
        pthread_mutex_unlock(&global_long_cas_lock);
        return true;
    }
    pthread_mutex_unlock(&global_long_cas_lock);
    return false;
#endif
```

**结论**：在 32 位系统上，`AtomicLong` 的 CAS 操作**不是真正的无锁**，而是退化为使用全局锁保护。这也是为什么 Android 早已全面转向 64 位的原因之一。

### 22.1 无锁事件总线（Event Bus）的简化实现

```kotlin
/**
 * 无锁事件总线实现
 * 
 * 设计要点：
 * 1. 使用 CopyOnWriteArrayList 存储订阅者（读多写少场景）
 * 2. 使用 AtomicReference 实现事件的"恰好一次"消费
 * 3. 避免在事件分发路径上使用任何锁
 */
class LockFreeEventBus {
    
    // 订阅者列表：使用 CopyOnWriteArrayList
    // 底层通过 ReentrantLock + 数组复制实现写时复制
    private val subscribers = CopyOnWriteArrayList<EventSubscriber>()
    
    // 粘性事件缓存：使用 ConcurrentHashMap
    // Key = 事件类型，Value = 最后一个事件
    private val stickyEvents = ConcurrentHashMap<Class<*>, Any>()
    
    // 事件序列号：用于去重
    private val eventSequence = AtomicLong(0)
    
    fun <T : Any> post(event: T) {
        val seq = eventSequence.incrementAndGet()
        val taggedEvent = TaggedEvent(event, seq)
        
        // 无锁遍历：CopyOnWriteArrayList 的迭代器不会抛 ConcurrentModificationException
        for (subscriber in subscribers) {
            if (subscriber.accepts(event::class.java)) {
                subscriber.onEvent(taggedEvent)
            }
        }
    }
    
    fun <T : Any> postSticky(event: T) {
        // ConcurrentHashMap.put 是线程安全的
        stickyEvents[event::class.java] = event
        post(event)
    }
    
    fun register(subscriber: EventSubscriber) {
        // CopyOnWriteArrayList.add 是线程安全的
        subscribers.add(subscriber)
        
        // 发送粘性事件
        for ((type, event) in stickyEvents) {
            if (subscriber.accepts(type)) {
                subscriber.onEvent(TaggedEvent(event, 0))
            }
        }
    }
    
    fun unregister(subscriber: EventSubscriber) {
        subscribers.remove(subscriber)
    }
    
    data class TaggedEvent<T>(val payload: T, val sequence: Long)
    
    interface EventSubscriber {
        fun accepts(eventType: Class<*>): Boolean
        fun onEvent(event: TaggedEvent<*>)
    }
}
```

### 22.2 无锁对象池（Object Pool）

```kotlin
/**
 * 基于 CAS 的无锁对象池
 * 
 * 使用场景：
 * - 高频创建/销毁的小对象（如 Message、Event）
 * - 避免 GC 压力
 */
class LockFreeObjectPool<T>(
    private val maxSize: Int,
    private val factory: () -> T,
    private val reset: (T) -> Unit = {}
) {
    // 使用 AtomicReferenceArray 存储对象
    private val pool = AtomicReferenceArray<T?>(maxSize)
    
    // 池中当前可用对象数量（近似值）
    private val size = AtomicInteger(0)
    
    // 获取/借出对象的指针
    private val takeIndex = AtomicInteger(0)
    
    // 归还对象的指针
    private val putIndex = AtomicInteger(0)
    
    /**
     * 从池中获取对象
     * 如果池为空，则创建新对象
     */
    fun acquire(): T {
        // 快速路径：尝试从池中获取
        repeat(maxSize) {
            val index = takeIndex.getAndIncrement() % maxSize
            val obj = pool.getAndSet(index, null)
            if (obj != null) {
                size.decrementAndGet()
                return obj
            }
        }
        
        // 慢速路径：池为空，创建新对象
        return factory()
    }
    
    /**
     * 归还对象到池中
     * 如果池已满，则丢弃对象（交给 GC）
     */
    fun release(obj: T) {
        reset(obj)  // 重置对象状态
        
        // 尝试放入池中
        repeat(maxSize) {
            val index = putIndex.getAndIncrement() % maxSize
            if (pool.compareAndSet(index, null, obj)) {
                size.incrementAndGet()
                return
            }
        }
        
        // 池已满，放弃（让 GC 回收）
    }
    
    /**
     * 预热池：创建初始对象
     */
    fun warmUp(count: Int = maxSize / 2) {
        repeat(count.coerceAtMost(maxSize)) {
            release(factory())
        }
    }
}

// 使用示例
class MessagePool {
    private val pool = LockFreeObjectPool(
        maxSize = 64,
        factory = { Message.obtain() },
        reset = { msg -> msg.recycle() }
    )
    
    fun obtainMessage(): Message = pool.acquire()
    fun recycleMessage(msg: Message) = pool.release(msg)
}
```

```c
Thread 1: takeIndex.getAndIncrement() -> 0  // 成功获取位置 0
Thread 2: takeIndex.getAndIncrement() -> 1  // 成功获取位置 1
Thread 3: takeIndex.getAndIncrement() -> 2  // 成功获取位置 2
Thread 4: takeIndex.getAndIncrement() -> 3  // 成功获取位置 3

// 每个线程在不同位置操作，几乎无 CAS 冲突！
// 这就是"分散热点"的设计思想

//       所有线程都会争抢 head 指针
```

```kotlin
            }
        } else {
            // 获取失败，降级处理或抛异常
            throw ConcurrentModificationException("Resource busy")
        }
    }
}
```

**核心认知链条**：

1. **CPU 层**：原子指令（CMPXCHG / LL/SC）是一切的基础，它保证了"比较并交换"这个复合操作的不可分割性。内存屏障确保操作的可见性和顺序性。

2. **JVM/ART 层**：`Unsafe` 类将 CPU 原子指令暴露为 Java API，同时隐藏了平台差异。ART 的 Intrinsic 机制确保关键方法被高效内联。

3. **Java API 层**：`AtomicInteger` 等类封装了 `Unsafe` 调用，提供类型安全和易用性。`AtomicStampedReference` 解决 ABA 问题。`LongAdder` 通过分段策略优化高并发性能。

4. **应用层**：开发者根据场景选择合适的原子类。低竞争用 `AtomicInteger`，高竞争计数用 `LongAdder`，需要版本控制用 `AtomicStampedReference`，多变量一致性回退到 `synchronized`。

**最终原则**：CAS 不是银弹。它是无锁编程的基石，但不是所有问题的最优解。理解其原理、了解其局限、根据场景选型——这才是架构师应有的技术判断力。
